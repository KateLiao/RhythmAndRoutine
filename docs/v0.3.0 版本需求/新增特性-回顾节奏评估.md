# Rhythm & Routine 回顾 · 记录后的节奏评估

> 版本：v0.3.0  
> 状态：已实现（v0.3.0）  
> 关联 Trellis 任务：`07-10-review-rhythm-evaluation`、`07-11-fix-review-rhythm-followups`
> 父任务：`07-09-v03-release`

## 背景与问题

产品设计（`docs/Rhythm & Routine 应用设计与开发.md`）已定义 Review：基于执行记录与节奏信号，输出目标进展、Routine 坚持、任务完成、日程偏差、Rhythm 模式与下一周期建议。

当前实现存在明显缺口：

1. 回顾页仍像演示壳：假 Pattern、日 Tab 误用本周指标、评估内容偏薄。
2. LLM 日/周同构，只喂瘦摘要，未突出用户自然语言补充感受。
3. `aggregateTaskStatus` 在任一关联日程块完成时就把 Task 标为完成，污染任务/目标进展评估。
4. 定时默认时间与「昨日/上周回顾」展示语义未对齐产品预期。

本需求把回顾升级为**可确认的周期节奏评估存档**，并与首页洞察错开职责。

## 产品目标

用户在日/周周期结束后，能读到一份有证据支撑的**评估正文**，并确认回顾本身与相关阶段节点；系统不替用户宣布 Task / Milestone / Outcome 已完成。

## 与首页的分工


| 表面  | 角色        | 用户看到的                 |
| --- | --------- | --------------------- |
| 首页  | 当前信号与即时行动 | 此刻建议 / 节奏发现 / 本周轨道    |
| 回顾页 | 周期评估存档    | 「昨日回顾」「上周回顾」评估正文 + 确认 |


回顾页**不做**「此刻建议」式即时改期主路径，也**不**在本版本自动产出 Adjustment ChangeSet。

## 范围

### 包含

1. **前置**：修正 Task ↔ ScheduleBlock 完成态关联（见下节）。
2. 日/周回顾分流：生成输入、输出 schema、展示语义。
3. 默认定时：每晚 23:00 日回顾；每周日 23:00 周回顾（用户时区）；展示为昨日/上周回顾直至下一份生成。
4. 保留手动重新生成（失败重试 / 补生成）。
5. 回顾页 UI 重设计：以评估正文为主，不详细罗列执行明细。
6. AI 失败时规则降级；生成幂等；用户确认/撤销确认回顾。

### 不包含

- 阶段回顾（goal-phase review）
- Agent 对话窗口体验优化（清单第 3 项）
- 首页洞察卡能力改造（除既有消费回顾 findings 的路径）
- Review → Adjustment ChangeSet
- 时长达标后自动完成 Task
- 回顾页内嵌完整执行账本

## 一、前置：Task 与 ScheduleBlock 完成语义

### 1.1 问题

当前任一关联日程块 `COMPLETED` → Task 直接 `COMPLETED`，把「一次会话完成」误当成「交付完成」。

### 1.2 规则


| 对象             | 含义          | 谁决定完成                              |
| -------------- | ----------- | ---------------------------------- |
| Schedule Block | 一次安排 / 一次投入 | 执行反馈即可                             |
| Task           | 可验收交付物      | **仅用户**在完成标准区确认（现有 `completeTask`） |


- 聚合状态不得因任一或全部块完成而把 Task 标为 `completed`。
- 有计划/进行中块 → `scheduled` / `in_progress`；有投入但未确认 → 保持推进态。
- 当累计真实投入 ≥ `estimatedMinutes`（或相关计划会话已结束）时，系统可提示「建议确认完成」，**不自动完成**。
- Task 与 ScheduleBlock 只通过 `taskId` / `ScheduleBlockTask.taskId` 关联，标题相同不产生关联。
- 通用 `PATCH /api/tasks/:id` 不接受 `status=completed`；唯一完成入口为 `POST /api/tasks/:id/complete`，且必须写入 `completionRecord`。
- 存量修复只处理 `completed AND completionRecord IS NULL` 的未归档 Task，按两类稳定 ID 关联的有效日程块重算非终态。脚本默认 dry-run，apply 前自动备份并使用版本条件更新；有用户确认记录的任务保持完成。

## 二、日回顾

### 2.1 定位

收工复盘：这一天执行得怎样、感受如何、第二天要不要轻调？

### 2.2 定时与展示

- 用户时区 **每天 23:00** 生成当天周期（00:00–23:00）的日回顾。
- Vercel Cron 负责部署环境的主动触发；若触发延迟或漏跑，下次触发会补生成最近已经到期的日回顾，不要求命中设置时间的精确单分钟。
- 数据库模式读取回顾列表时执行同一套到期同步；本地 `next dev` / `next start` 还会在进程内每 5 分钟自动同步一次，无需依赖外部 Cron。
- 手动「生成 / 重新生成」默认作用于最近到期周期或当前展示中的那份回顾，禁止在设置时间前把日历「今天」当成已到期日回顾。
- 回顾页优先展示最近到期周期；尚未到期的抢先生成本身不会盖住昨日回顾。
- 自动恢复一次只检查并补最近一份到期日回顾，不批量回填更早历史缺口；已有非失败周期记录跳过，失败记录可重试。
- 2026-07-14 / 07-16 修复验收时，已对确认漏跑的日回顾执行一次性幂等补生成。
-  过了 0 点，在下一份日回顾生成前，回顾页日 Tab 标题为 **「昨日回顾」**。
- 默认设置：`dailyReviewTime=23:00`。

### 2.3 生成时 LLM 输入

**共用**：`period`、`productConstraints`、`periodMetrics`

**日回顾专有**：


| 字段                      | 说明                                                                                                                          |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `todayBlocks[]`         | 每个日程的执行详情（时间、status、planned/actualMinutes、result、tags、comfortable、timeFit、quality、obstacle、deviationReason、nextAction、关联标题） |
| 用户 `note` / 补充感受        | **若有，须在 prompt 中要求着重参考**，不得被标签统计淹没                                                                                          |
| `pendingFeedbackCount`  | 已结束未记录执行的块数                                                                                                                 |
| `activeRhythmSignals[]` | 有效信号摘要，最多约 5 条                                                                                                              |


不含：全量 Goal/Milestone 树、跨日任务累计、上周对比。

### 2.4 页面呈现

1. 页面级「日回顾 / 周回顾」切换位于 Hero 外；页头：（今日）昨日回顾 + 日期 + 状态；主视觉卡片直接展示该份日回顾 `summary` 或日专属空态/生成态文案，不使用固定标语。
2. 轻量指标条（辅助，不占主视觉）。
3. **评估正文（主内容）**：以「关键发现」与「建议」为左侧主报告。
4. 补充观察（右侧栏）：执行亮点、节奏解读、下一步轻量建议等可选区块。
5. 操作：确认 / 重新生成 / 请小律解释。

❌ **不**逐条展示昨日执行明细（明细留在日历/任务侧）。

## 三、周回顾

### 3.1 定位

节奏与目标校准：上周节奏如何、任务/Routine 进展如何、哪些阶段该由你确认？

### 3.2 定时与展示

- 用户时区 **每周日 23:00** 生成当周周期（与现有周边界一致）。
- Cron 延迟或漏跑后会按用户设置补生成最近已经到期的周回顾；读取回顾列表与本地进程内定时器也会执行相同自愈。
- 手动生成 / 重新生成周回顾同样绑定最近到期周或当前展示周，不在触发日前提前把「本周」当作已到期周回顾。
- 自动恢复一次只检查并补最近一份到期周回顾，不批量生成更早历史周；已有非失败周期记录跳过，失败记录可重试。
- 在下一周周一 0 点之后，下一份周回顾生成前，周 Tab 标题为 **「上周回顾」**。
- 默认：`weeklyReviewDay=0`（周日），`weeklyReviewTime=23:00`。

### 3.3 生成时 LLM 输入


| 字段                            | 说明                                                             |
| ----------------------------- | -------------------------------------------------------------- |
| `weekBlocksSummary`           | 按日聚合或代表块；**带 note 的块优先保留原文摘要**                                 |
| `taskProgress[]`              | 本周有活动的 Task（含 estimatedMinutes、投入、`readyForCompletionSuggest`） |
| `routineStability[]`          | Routine 坚持情况                                                   |
| `scheduleDeviation`           | 偏差计数与高频原因                                                      |
| `goalProgressHints[]`         | 活跃 Goal 投入 + 待检查 Milestone/Outcome                             |
| `activeRhythmSignals[]`       | 有效信号                                                           |
| `recentDailyReviewFindings[]` | 本周日回顾 findings（若有）                                             |

#### 3.3.1 周回顾上下文压缩

周回顾**禁止**把整周逐条执行原文塞进模型。服务端先做确定性压缩，再生成：

1. **日回顾作先验**：有足够日回顾时，以去重后的日 findings + 7 日聚合指标为主，不再喂该日逐块明细。
2. **数字规则算死**：完成率、投入、标签分布、Top 偏差原因由代码计算，模型只解释。
3. **note 优先 / 异常优先**：用户补充感受与 missed/阻力摘录进有限名额池（约 ≤12 条，每条短截断）；常规完成块只进计数。
4. **实体裁剪**：仅本周有活动的 Task / Routine / Goal 进入列表，并设条数上限。
5. **单跳生成**：不做「先摘要再总结」的二次 LLM。

硬预算与有/无日回顾的分支见 Trellis `design.md` §3.3.1。

### 3.4 页面呈现

1. 页面级「日回顾 / 周回顾」切换位于 Hero 外；页头：上周回顾 + 周区间 + 状态；主视觉卡片直接展示该份周回顾 `summary` 或周专属空态/生成态文案。
2. 轻量周指标。
3. **评估正文**：主报告聚焦「关键发现」「节奏解读」「本周建议」「下周建议」。
4. **校准侧栏**：独立承载任务进展、Routine 坚持与目标检查；任务汇总同时覆盖主 `taskId` 与 `ScheduleBlockTask` 多任务关联并按 ID 去重。
5. 「只由你确认」：Milestone / Outcome / 建议确认完成的 Task，紧随校准信息出现。
6. 操作：确认 / 重新生成 / 请小律解释（无 ChangeSet 主按钮）。

❌ **不**逐条罗列本周执行明细。

## 四、共用输出 Schema

日/周共用一套增强结构（日简填、周填满）：


| 字段                                                            | 必填  | 日       | 周                    |
| ------------------------------------------------------------- | --- | ------- | -------------------- |
| `summary` / `findings` / `suggestions` / `source`             | 是   | ✓       | ✓                    |
| `sessionHighlights`                                           | 否   | ✓       | 可空                   |
| `rhythmNotes`                                                 | 否   | 轻量      | ✓                    |
| `taskProgressNotes` / `routineNotes` / `goalCheckSuggestions` | 否   | 通常空     | ✓                    |
| `nextCycleSuggestions`                                        | 否   | 今晚/明天轻调 | 下周建议（文案，非 ChangeSet） |


`source`：`ai` | `rules`。系统不得在输出中宣布 Task / Milestone / Outcome 已完成。

### 4.1 报告语言原则

回顾报告是给用户看的自然中文产品文案，不是数据审计、JSON 解释或调试日志：

- 必须使用第二人称「你」，不得使用「用户」「该用户」「这位用户」等第三人称称谓。
- 用户可见内容不得暴露数据库/API/JSON 字段名、英文枚举值、布尔值、代码式键值对或字段名括注。
- 输入中的 `tags`、`comfortable`、`timeFit`、`quality`、`status`、`result`、`smooth`、`good`、`great` 等仅作为内部证据读取，输出时必须改写成自然中文，例如「反馈较顺畅」「时间安排比较匹配」「质量反馈很好」「执行中有阻力」。
- 前端展示层保留兜底清洗，避免已有旧回顾或模型偶发漏出内部参数。

## 五、约束与降级

- 执行记录发生在日程结束后；`planned` 过期 ≠ 未开始。
- 不得捏造数据；区分事实与判断。
- AI / 结构化输出失败 → 规则引擎降级，仍生成可读回顾。
- Cron、读取时自愈、本地进程内定时器与手动生成共用同一生成服务；幂等键保持周期唯一。
- Cron、读取时自愈与本地定时器只同步最近已经到期的日、周周期，每种最多一份；已有非失败记录不重复生成，失败记录允许重试。
- 手动重新生成优先重写当前展示周期；空态时生成最近到期周期，而不是尚未到期的日历今天 / 本周。

## 六、验收标准

1. 完成一个关联日程块不会把 Task 标为 `completed`；仅用户确认才会；达标可「建议确认」而不自动完成。
2. 日回顾在用户时区 23:00 到期；Vercel Cron、本地进程内定时器或数据库模式读取列表时，均可幂等补生成最近到期周期；至下一份日前，日 Tab 为「昨日回顾」。
3. 周回顾在周日 23:00 到期；Vercel Cron、本地进程内定时器或数据库模式读取列表时，均可幂等补生成最近到期周期；至下一份周回顾前，周 Tab 为「上周回顾」。
4. 存在用户补充 note 时，评估 findings 能反映其内容（非仅标签计数）。
5. 持久化符合增强 schema；回顾页以评估正文为主，**不**详细列出周期内每条执行记录。
6. 可确认/撤销确认回顾；失败可手动重新生成；规则降级可读。
7. 周回顾可展示 Milestone/Outcome（及建议确认 Task）入口，系统不自动标完成。
8. 回顾页无「此刻建议」主路径，不自动产出 Adjustment ChangeSet。
9. 左侧「今天」badge 等于用户时区当天 `planned` / `in_progress` 块数，0 时隐藏；过期但未记录结果的块仍计入。
10. 左侧「回顾」仅在至少一份回顾为 `awaiting_confirmation` 时显示「新」，进入页面不清除，确认后由真实状态自动消失。
