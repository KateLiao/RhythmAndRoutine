
# Rhythm & Routine 首页右侧三张卡片产品需求

## 背景

当前首页右侧有三张卡片，现有内容为占位文案。需要重新设计这三张卡片的信息架构，使其更符合 Rhythm & Routine 的核心价值：

> 系统不是单纯展示日程，而是基于用户真实执行记录，逐渐理解用户的节奏，并给出更适合后续安排的建议。

首页主区域已经承担「今天安排了什么、当前日程是什么」的展示功能，因此右侧卡片不应重复日程列表信息，而应承担「节奏判断、执行洞察、目标校准」的作用。

右侧三张卡片需要形成清晰分工：

1. 此刻建议：现在 / 今天接下来该怎么做。
2. 节奏发现：系统最近从执行记录中理解了用户什么。
3. 本周轨道：本周整体负荷和目标推进是否健康。

整体信息层级为：

Action → Insight → Progress

即：

1. 我现在该怎么做？
2. 系统为什么这样理解我？
3. 我这一周整体有没有偏航？

## 一、卡片 1：此刻建议

### 1.1 定位

卡片 1 是「即时行动建议卡」。

它回答的问题是：

> 我现在 / 今天接下来最适合做什么？

它是短周期、即时决策、强行动导向的卡片。

### 1.2 用户价值

用户打开首页时，不一定想先看数据，而是想知道：

- 我现在应该继续原计划吗？
- 接下来最适合做什么？
- 要不要调整今天后面的安排？
- 当前节奏是否适合进入深度任务？
- 是否应该先休息、缓冲，或者做低阻力任务？

### 1.3 内容结构

卡片应包含：

1. 卡片标题
2. 当前判断
3. 推荐下一步
4. 一句话原因
5. 操作按钮

### 1.4 推荐文案结构

```text
此刻建议

下午适合稳稳推进

你上午已经完成 1 个高专注任务。
接下来建议先做 30 分钟低阻力任务，再进入下一个深度任务。

推荐下一步：
13:30 英语口语练习

[接受安排] [换个建议]


### 1.5 数据来源

卡片内容可以基于以下信息生成：

* 当前时间
* 今日已完成日程块
* 今日剩余日程块
* Schedule Block 的执行状态
* 用户近期 Rhythm Signal
* 当前任务类型
* 今日负荷情况
* 是否存在连续高强度任务
* 是否存在未完成 / 延期任务

### 1.6 展示原则

这张卡必须是「行动导向」的。

应该避免只展示泛泛的鼓励，例如：

```text
今天也要一步一步前进。
```

也应该避免只展示总结，例如：

```text
你今天已经完成了 2 个任务。
```

更好的表达应该是：

```text
你上午已经完成 1 个高专注任务。接下来建议先做一个低阻力任务，避免连续高强度消耗。
```

### 1.7 按钮建议（v0.3 已确认：轻量调整 + 候选轮换）

主按钮按建议类型直接执行日程变更，**不走 ChangeSet**（见上表）。

**「换个建议」（Phase B）**：

* 首次生成时 LLM 一次产出主建议 + `alternateCandidates[]`（2–4 条），入库；用户点击仅 **轮换下标**，不再调 LLM。
* `alternateIndex` 同步落库，刷新后可恢复当前查看的第几条。
* 候选用尽后展示 **exhausted 空状态**（与 §8.1「数据不足」区分）：

```text
本轮建议已经看完。先按当前安排推进，完成或记录一次执行后，系统会结合新数据生成下一批建议。
```

MVP（Phase A）阶段仍为规则池 `alternateIndex` 轮换，逻辑与上一致。

## 二、卡片 2：节奏发现

### 2.1 定位

卡片 2 是「节奏模式洞察卡」。

它回答的问题是：

> 系统最近从我的执行记录中发现了什么长期规律？

它不是用于告诉用户「现在该干什么」，而是让用户感受到：

> 系统正在通过真实数据理解我的节奏。

### 2.2 与卡片 1 的区别

卡片 1 是即时行动建议：

> 我现在 / 今天接下来该怎么做？

卡片 2 是长期节奏洞察：

> 系统最近理解了我什么？

二者不能重复。

| 卡片   | 核心问题         | 时间尺度           | 用户感受    |
| ---- | ------------ | -------------- | ------- |
| 此刻建议 | 我现在该怎么做？     | 此刻 / 今天        | 被陪伴、被提醒 |
| 节奏发现 | 系统最近理解了我什么？  | 近 7 天 / 近 14 天 | 被看见、被理解 |
| 本周轨道 | 我的目标推进还在正轨吗？ | 本周 / 本阶段       | 被校准、被管理 |

### 2.3 卡片名称

建议使用：

```text
节奏发现
```

不建议继续使用「小律观察」，因为这个名称容易和第一张「此刻建议」混淆，都像是 AI 在给建议。

其他可选名称：

* 最近的节奏
* 你的节奏模式
* Rhythm Insight
* 系统学到的你

优先推荐：

```text
节奏发现
```

### 2.4 内容结构

卡片应包含：

1. 卡片标题
2. 发现内容
3. 数据证据
4. 对后续安排的影响
5. 操作入口

### 2.5 推荐文案结构

```text
节奏发现

你最近更容易在上午进入深度工作。

证据：
过去 7 天，10:00-12:00 的深度任务完成率最高；
20:00 后的写作任务有 3 次延期。

这个发现会用于之后的日程安排：
深度任务将优先推荐在上午。

[查看详情] [用于下次安排]
```

### 2.6 重要原则

卡片 2 应该少写「建议你立刻做什么」，多写「系统发现了什么」。

不推荐：

```text
建议你把深度任务放到上午。
```

推荐：

```text
这个发现会影响之后的自动安排：深度任务将优先推荐在上午。
```

这样可以避免和卡片 1 重复。

### 2.7 数据来源

卡片内容可以基于最近 7 天 / 14 天的数据生成：

* Schedule Block 完成率
* 延期次数
* 跳过次数
* 改期次数
* 执行时间段
* 任务类型
* Routine 执行情况
* Rhythm Feedback 标签
* 用户自然语言反馈
* 实际耗时与预计耗时偏差

### 2.8 可识别的节奏发现类型

MVP 阶段可以先支持规则型发现，不需要复杂模型。

可支持以下类型：

1. 高完成率时间段

```text
你最近在上午 10:00-12:00 的深度任务完成率最高。
```

2. 低完成率时间段

```text
你最近安排在 20:00 后的写作任务更容易延期。
```

3. 某类任务启动困难

```text
你连续 3 次推迟了英语自由对话练习。
```

4. 某类任务适合低门槛开始

```text
你在先做 10-15 分钟小任务后，更容易进入后续深度任务。
```

5. 训练 / 外出后不适合高认知任务

```text
你在力量训练后的高认知任务完成率较低。
```

6. Routine 压力过高

```text
某个 Routine 连续多次被跳过，可能需要降低频率或门槛。
```

### 2.9 按钮建议

按钮可以包括：

* 查看详情
* 用于下次安排
* 忽略这个发现

MVP 阶段可以先实现：

* 主按钮：用于下次安排
* 次按钮：查看详情

点击「用于下次安排」后，可以将该 Rhythm Signal 标记为 active，用于后续 Agent 排程逻辑。

## 三、卡片 3：本周轨道

### 3.1 定位

卡片 3 是「周状态与目标推进卡」。

它回答的问题是：

> 本周整体安排和执行是否健康？目标推进有没有偏航？

它不是当下建议，也不是节奏发现，而是更宏观的状态面板。

### 3.2 用户价值

用户需要知道：

* 本周整体负荷是否过高？
* 本周安排是否过松？
* 时间是否投入到了重要目标上？
* 哪个目标投入不足？
* 哪些 Routine 稳定，哪些正在掉线？
* 是否需要在周末补进度？
* 是否需要留恢复空间？

### 3.3 卡片名称

建议使用：

```text
本周轨道
```

比「本周负荷」更符合产品定位。

「本周负荷」只强调忙不忙，而「本周轨道」能同时表达：

* 负荷
* 节奏
* 目标投入
* 是否偏航

其他可选名称：

* 本周状态
* 目标推进
* 本周节奏
* 本周负荷

优先推荐：

```text
本周轨道
```

### 3.4 内容结构

卡片应包含：

1. 卡片标题
2. 当前状态判断
3. 本周已安排 / 已完成时间
4. 目标投入提醒
5. 风险或建议

### 3.5 推荐文案结构

```text
本周轨道

节奏适中

已安排 16h，完成 8.5h。
Vibe Coding 推进稳定，但英语练习低于计划。

建议：
本周剩余时间补 2 次 15 分钟英语练习。
```

### 3.6 数据来源

卡片内容可以基于：

* 本周 Schedule Block 总计划时长
* 本周已完成时长
* 本周延期 / 跳过任务数
* 各 Goal / Project / Skill / Routine 投入时间
* Routine 完成率
* 深度任务与低阻力任务比例
* 恢复时间占比
* 用户本周重点目标
* Review 结果

### 3.7 状态判断

可以设置以下状态：

```text
偏松
适中
适中偏满
过载
偏航
```

对应逻辑示例：

* 偏松：本周计划时间明显低于用户可投入时间，且目标进展不足。
* 适中：计划负荷合理，完成情况稳定。
* 适中偏满：计划较满，但仍有恢复空间。
* 过载：连续日程过密，延期或跳过明显增加。
* 偏航：时间投入与当前重点目标不匹配。

### 3.8 展示原则

卡片 3 不要只是展示时间统计。

不推荐：

```text
已安排 16 小时，完成 8 小时 30 分。
```

推荐：

```text
已安排 16h，完成 8.5h。
Vibe Coding 推进稳定，但英语练习低于计划。
```

这张卡必须把「时间统计」和「目标推进」联系起来。

## 四、三张卡片整体关系

### 4.1 信息层级

右侧三张卡片的整体顺序为：

```text
1. 此刻建议：现在怎么做
2. 节奏发现：系统学到了什么关于我
3. 本周轨道：整体有没有偏航
```

### 4.2 三张卡片的核心分工

| 卡片   | 核心问题      | 时间尺度       | 决策类型   |
| ---- | --------- | ---------- | ------ |
| 此刻建议 | 现在该怎么做？   | 此刻 / 今天    | 即时行动决策 |
| 节奏发现 | 系统理解了我什么？ | 近 7 / 14 天 | 规律认知   |
| 本周轨道 | 本周是否在正轨？  | 本周 / 本阶段   | 目标校准   |

### 4.3 用户心理感受

三张卡片合起来要让用户感受到：

```text
系统不仅知道我今天排了什么，
还知道我最近是怎么执行的，
哪里容易偏航，
现在应该怎么调，
以及这些调整如何帮助我接近目标。
```

## 五、避免的问题

### 5.1 避免卡片 1 和卡片 2 重复

卡片 1 不要做长期规律总结。

卡片 2 不要做即时行动建议。

错误示例：

```text
卡片 1：建议你下午做低阻力任务。
卡片 2：建议你以后下午也做低阻力任务。
```

正确区分：

```text
卡片 1：现在建议你先做 30 分钟低阻力任务。
卡片 2：系统发现你最近在午后缓冲 30 分钟后，后续任务完成率更高。
```

### 5.2 避免鸡汤化

不应出现大量泛泛鼓励，例如：

```text
今天也要好好努力。
一步一步就会越来越好。
```

可以有温度，但必须基于数据或具体上下文。

### 5.3 避免纯数据化

不应只展示：

```text
完成率 67%
延期 3 个
安排 16 小时
```

必须解释这些数据意味着什么，以及用户可以如何调整。

### 5.4 避免重复主区域信息

主区域已经展示今天的日程块。

右侧卡片不应该重复展示：

* 今天有哪些任务
* 每个任务几点开始
* 每个任务状态

右侧应该解释：

> 这些日程和执行记录意味着什么。

## 六、MVP 实现建议

### 6.1 第一阶段：静态规则 + 真实数据

MVP 不需要复杂 AI 推理，可以先用简单规则生成卡片。

例如：

#### 此刻建议规则

* 如果上午已完成高专注任务，则下午建议低阻力任务或缓冲。
* 如果当前时间已过某个任务计划开始时间，但任务未开始，则提醒是否调整。
* 如果今天剩余任务过多，则建议减少或延后部分任务。
* 如果连续两个高认知任务相邻，则建议插入休息。

#### 节奏发现规则

* 统计最近 7 天不同时间段任务完成率。
* 统计不同任务类型的延期率。
* 统计 Routine 跳过次数。
* 统计计划耗时与实际耗时偏差。
* 统计某些任务是否连续延期。

#### 本周轨道规则

* 统计本周计划总时长、完成总时长。
* 统计各目标投入时间。
* 判断当前重点目标是否投入不足。
* 判断 Routine 完成率是否下降。
* 判断延期 / 跳过任务是否过多。

### 6.2 第二阶段：接入 LLM 生成自然语言（Phase B 架构已确认）

**混合合成（方案 C）**：

* **此刻建议**：独立 `generateMomentInsight`，含 `alternateCandidates` 供轮换。
* **节奏发现 + 本周轨道**：单次 `generateSlowInsight`，一次输出两卡。
* 固定 workflow（`generateObject`），**不走 Agent Loop**。
* 快照落库 `HomeInsightSnapshot`（`kind=moment|slow`），Web 刷新读**最新一条**记录。
* **本地模式（无 DB）**：不调 LLM，沿用 Phase A 规则引擎。

规则仍负责 Facts 与结论；LLM 负责组织表达与 `proposedChange` 结构化动作。

**执行记录约束（2026-07-09）**：用户不在任务开始时打卡，只在结束后「记录执行」复盘。禁止建议「改到此刻开始」；优先 `open_execution_feedback`。

#### 6.2.1 刷新策略（Phase B.1，2026-07-09 变更）

> 废弃 `factsHash` / TTL / 写操作 invalidate 触发的自动重算（不稳定，刷新易掉回规则建议）。

| 卡片 | 定时生成 | 手动更新 |
|------|----------|----------|
| 此刻建议 | **每小时整点**（用户时区 `:00`）入库 | 卡片「更新」按钮 |
| 节奏发现 + 本周轨道 | **每周 2 次**：周三 **08:00**、周日 **20:00**（用户时区）入库 | 两卡各自「更新」按钮（实际触发同一条 `slow` 生成） |

**读取**：`GET /api/home/insights` 仅返回各 `kind` **最新**快照；不在 GET 时因数据漂移自动重算。若尚无快照，可冷启动生成一次。

**存储**：快照追加写入；定期清理过久历史（moment 7天/168条，slow 180天/32条，取较严）。

**上下文**：slow / moment 生成时注入的 Facts 与 prompt 与 Phase B 基本一致。

**换个建议**：仍轮换 `alternateCandidates`，不调 LLM。

**实现状态**：

* Phase B.1 已落地：定时 + 手动刷新；GET 只读最新快照；历史按保留策略清理。
* 卡片来源标签：「小律生成 / 规则建议」；展示相对更新时间。

### 6.3 第三阶段：将建议结构化

后续需要将建议保存为结构化对象，以支持北极星指标统计。

建议对象字段：

```text
Suggestion
- id
- suggestion_type
- source_card
- target_object_type
- target_object_id
- reason
- evidence
- proposed_change
- user_response
- applied_at
- execution_result
- evaluated_at
```

其中 source_card 可以是：

```text
moment_suggestion
rhythm_insight
weekly_track
```

## 七、验收标准

### 7.1 卡片 1：此刻建议

验收标准：

* 能基于当前时间、今日日程和已完成情况生成一条即时建议。
* 建议必须包含一个明确的下一步行动。
* 建议不能只是鼓励性文案。
* 建议不能重复展示日程列表。
* 至少有一个可点击操作按钮。

### 7.2 卡片 2：节奏发现

验收标准：

* 能基于最近 7 天或 14 天执行记录生成一个节奏发现。
* 内容必须包含「发现」和「证据」。
* 文案应弱化即时建议，强调系统理解到的规律。
* 发现可以用于后续日程安排。
* 不应和卡片 1 的即时建议重复。

### 7.3 卡片 3：本周轨道

验收标准：

* 能展示本周整体状态判断。
* 能展示本周安排时间与完成时间。
* 能指出至少一个目标投入或负荷相关的问题。
* 不应只是纯时间统计。
* 能让用户判断本周是否需要调整计划。

## 八、默认空状态

如果当前数据不足，三张卡片需要有合理空状态。

### 8.1 此刻建议空状态

```text
此刻建议

先从一个小时间块开始

你今天还没有足够的执行记录。
可以先安排一个 30 分钟的小任务，让系统开始了解你的节奏。
```

### 8.2 节奏发现空状态

```text
节奏发现

完成几次执行反馈后，这里会出现你的节奏规律。

例如：
你更适合在哪些时间段做深度任务，
哪些 Routine 容易被跳过，
哪些任务需要拆得更小。
```

### 8.3 本周轨道空状态
