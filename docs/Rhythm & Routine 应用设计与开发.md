## 2. 核心概念

### 2.1 Rhythm 定义
任务完成效率与个人状态密切相关。当用户进入心流状态时，思考与行动能够持续推进，任务完成节奏通常更快、更连贯，也更具推进感；而在注意力分散、动力不足或状态欠佳时，行动节奏则会放缓，甚至出现停滞。
这种由行为模式所体现出的内在节奏（Rhythm），往往是客观存在的，但用户未必能够清晰地感知和描述。通过对日常行为和任务执行过程的持续记录与分析，系统可以帮助用户识别自身的节奏规律，从而找到更适合自己的工作、学习和生活方式。

### 2.2 Routine 定义
有些行为的价值，并不体现在一次完成的结果上，而体现在长期、持续的重复之中。虽然重复本身往往是枯燥的，也很容易因为短期看不到回报而在某一天被放弃，但正是这些看似微小的日常习惯，会随着时间积累产生复利效应。
例如：
- 早起后不立即查看手机；
- 每天练字；
- 坚持吉他爬格子练习；
- 固定时间阅读或运动。
这些行为未必能够在短期内带来显著变化，却可能成为影响一生的长期资产。
Routine 并非简单的待办事项，而是能够持续重复、并与长期目标形成因果联系的行为模式。通过将日常坚持的 Routine 与目标之间的关联关系可视化，系统能够帮助用户看到每一次重复背后的长期价值，让原本难以感知的成长过程变得可见，从而理解复利是如何在日复一日的行动中逐渐产生，并获得持续坚持下去的动力。

### 2.3 Rhythm 与 Routine 的关系
Routine 负责长期重复的行动系统，帮助用户通过持续练习、稳定执行和复利积累，逐步接近长期目标。

Rhythm 负责动态适配真实的人。它不是只记录“用户做了什么”，而是持续理解用户在什么时间、什么状态、什么任务类型下推进得更好，并据此调整后续计划。

因此，Routine 让用户持续行动，Rhythm 让系统理解用户真实的行动节奏，并基于执行反馈动态调整计划。

### 2.4 Rhythm 的动态调整机制
Rhythm 可以被理解为系统中的反馈控制机制：

1. 记录节奏信号：在每个日程块或任务执行后，记录精力、专注度、启动难度、完成质量、实际耗时、情绪状态、是否进入心流等信号。
2. 识别节奏模式：系统逐渐识别用户在哪些时间段适合深度工作，哪些时间段适合运动、整理、阅读或低阻力任务。
3. 评估计划偏差：当计划没有完成时，系统不只判断“是否完成”，还要分析偏差原因，例如任务过大、时间估算错误、状态不匹配、安排过密、目标优先级变化等。
4. 动态调整安排：系统根据节奏模式和执行偏差，重新建议任务时间、任务粒度、Routine 频率、当天负荷和恢复安排。

常见调整方式包括：
1. 将高认知任务安排到高能量窗口。
2. 将低阻力任务安排到低状态时段。
3. 对连续失败的 Routine 降低门槛。
4. 在状态稳定时提高挑战强度。
5. 在负荷过高时主动加入恢复时间。
6. 当目标推进偏离预期时，重新拆解任务或调整优先级。

## 3. 目标用户与核心定位

### 3.1 种子用户
种子用户首先是我自己。这个项目是一个比较私人的 AI Native Agent 项目，首先要解决我在 Gap 过程中的日程安排和目标跟踪问题。

在 Gap 阶段，我拥有较多可以自主支配的时间，但也因此缺少外部结构、节奏约束和持续反馈。系统需要帮助我把长期目标拆解成可执行任务，再把任务落到具体日程中，并在执行过程中持续记录、评估和调整计划，避免空闲时间被碎片化消耗。

### 3.2 扩展目标群体
从我自己扩散出去，目标用户可以是和我类似的人：他们拥有较多自主时间，需要通过系统来完成空闲时间的日程安排，并希望通过日程安排观察自己距离设定目标还有多远。

这类用户通常不是单纯“任务很多”，而是面临以下问题：
1. 目标重要但周期较长，短期反馈不明显。
2. 时间相对自由，但容易失焦或被碎片化消耗。
3. 缺少外部节奏约束，需要自己建立稳定的行动系统。
4. 计划经常因为状态、环境、优先级变化而失效。
5. 希望 AI 不只是提醒，而是能够基于真实执行情况动态评估目标进度，并调整后续计划。

### 3.3 核心定位
Rhythm & Routine 是一个 AI Native 的个人目标推进系统，帮助拥有自主时间的人，把长期目标转化为可执行日程，并基于真实执行记录持续评估、调整和陪伴用户达成目标。

它的关键差异点在于：目标不是静态清单，日程不是孤立安排，AI 也不是单纯的聊天助手，而是一个持续参与目标推进的计划伙伴。

## 4. 产品设计

### 4.1 用户核心旅程
**刚进入系统**
1. 确定自己要达成的目标
2. 为达成目标拆解任务
3. 设定任务执行计划
**执行计划阶段**
4. 通过日程系统记录计划执行情况，查看具体日程
5. 评估是否需要调整执行计划
**回顾阶段**
6. 评估任务完成情况
7. 回顾目标，调整拆解下一阶段任务

### 4.2 实体设计
目标、任务、日程的分类

#### 4.2.1 实体设计原则
实体设计不应把所有内容都塞进 Goal。对于 Rhythm & Routine 来说，用户的 Gap 设想里同时包含了想达成的结果、想养成的习惯、想完成的项目、想积累的能力、想消费的资源，以及真实执行过程中的状态反馈。

因此，实体模型应围绕三层展开：

1. 方向层：回答用户最终想去哪里。
2. 推进层：回答用户通过什么方式持续接近目标。
3. 现实层：回答用户在真实生活中如何安排、执行、记录和调整。

#### 4.2.2 系统可观测边界与确认原则
系统不应该替用户证明目标已经达成，而是通过组织计划、记录执行、呈现历程，帮助用户在回顾中判断自己离目标还有多远。

系统能够较可靠记录的是：
1. 用户计划了什么。
2. 用户是否执行了这些计划。
3. 计划是否被改期、跳过或取消。
4. 用户在执行后的轻量 Rhythm 反馈。
5. 用户在回顾中对目标进展的自我判断。

因此，Task 和 Schedule Block 是系统可以直接记录的对象；Milestone 和 Outcome 则不应该由系统自动推理达成，而应该由用户在 Review 中确认。

```text
Task / Schedule Block
  -> 由执行行为更新状态

Milestone / Outcome
  -> 由 Review 中的用户确认更新状态
```

对健身目标来说，系统不需要记录训练动作、组数、重量、饮食和睡眠，也不需要替用户判断体型是否达标。系统只需要呈现：
1. 过去一段时间是否保持了每周 3-4 次训练。
2. 推 / 拉 / 腿是否大致均衡。
3. 哪些时间段更容易完成训练。
4. 哪些训练安排经常被跳过或改期。
5. 用户在回顾时认为体型和训练能力是否正在推进。

对 Vibe Coding 目标来说，系统不需要自动判断“用户是否已经成为 AI Agent 开发工程师”，也不需要自动判定某个交付物是否合格。系统只需要呈现：
1. 项目拆出了哪些阶段性目标。
2. 每个阶段下完成了哪些 Task。
3. 哪些任务被推迟最多。
4. 哪些类型的任务最容易完成。
5. 用户在回顾时确认 Milestone 是否完成，以及能力是否有所推进。

这个原则可以概括为：

> 系统不追求客观判定一切，而是通过持续记录行动历程，帮助用户更清楚地看见自己和目标之间的距离。

#### 4.2.3 方向层实体
**Goal 目标**  
回答“我最终想达成什么变化”。Goal 更像方向和承诺，不一定直接等同于一个具体任务。

例如：
- 提升身体状态。
- 提升英语面试表达能力。
- 成为更成熟的 AI Agent 产品构建者。
- 做出 Rhythm & Routine 的可用 MVP。

**Outcome 结果指标**  
回答“如何判断目标达成了”。Outcome 让 AI 可以评估用户距离目标还有多远。

例如：
- 体脂控制到 18-19，有清晰的 4 块腹肌，腿部肌肉明显。
- 面试时能够较为流利地讲述自己的项目经历和对 AI 的理解。
- 完成一个可用的 AI Native Agent 产品原型。
- 完成 Outer Wilds 指弹，能够弹唱五月天的几首歌。

#### 4.2.4 推进层实体
**Project 项目**  
回答“哪些事情需要阶段性产出”。Project 通常有明确交付物或作品。

例如：
- 以“成年后的暑假”为主题拍摄纪录片。
- 构建健身相关产品。
- 构建英语学习相关产品。
- 构建 Rhythm & Routine MVP。

**Skill 能力**  
回答“我想长期变强的能力是什么”。Skill 不一定有明确完成点，但可以通过练习记录、作品产出和阶段评估来追踪成长。

例如：
- 英语口语表达。
- AI Agent 开发。
- 肌肥大、增力训练周期和单次训练计划制定。
- 画画、吉他、产品设计。

**Routine 长期重复行为**  
回答“哪些行为需要周期性重复”。Routine 不是简单 habit，而是与长期目标Goal形成因果关系的复利行为。

例如：
- 每天控笔练习 20 分钟。
- 每周 3-4 次力量训练。
- 每天英语口语练习。
- 固定阅读并输出读书笔记。

**Task 行动任务**  
回答“下一步具体做什么”。Task 是一次性、可完成、可验收的行动，每一个 Task 都应该推动用户往自己的某一个 Goal 更近一步。

例如：
- 设计本周训练计划。
- 读完 The Mom Test 第一章。
- 整理一版 AI 产品经理作品集。
- 实现一个日历原型。

#### 4.2.5 现实层实体
**Schedule Block 日程块**  
回答“什么时候做”。Schedule Block 是目标、项目、能力、Routine 和任务落到真实生活里的交汇点。

例如：
- 周二 19:00-20:30 腿部训练。
- 今天 14:00-16:00 做 Rhythm & Routine 实体设计。
- 明天 10:00-11:00 英语项目经历口述练习。

**Execution Record 执行记录**  
回答“实际做得怎么样”。它记录计划与实际之间的差异。

可记录的信息包括：
- 是否完成。
- 实际耗时。
- 完成质量。
- 遇到的阻碍。
- 偏差原因。
- 下一步处理建议。

**Rhythm Feedback 节奏反馈**  
回答“这次执行体验如何”。它是用户对一次 Schedule Block 执行体验的轻量反馈，也是 Rhythm 负责动态调整的关键输入。

可记录的信息包括：
- 感受标签：顺畅、有阻力、勉强完成、状态很好、状态很差、被打断、没开始。
- 可选自然语言补充。
- 执行过程是否舒适。
- 是否存在明显阻力。
- 当前时间段是否适合这类任务。

**Rhythm Signal 节奏信号**  
回答“系统从持续反馈中识别出了什么模式”。Rhythm Signal 不是用户每次都需要填写的复杂数据，而是 Agent 基于 Execution Record、Rhythm Feedback 和 Review 提取出的节奏判断。

可提取的信息包括：
- 哪些时间段更适合哪类任务。
- 哪些任务经常启动困难。
- 哪些任务粒度过大。
- 哪些 Routine 近期压力过高。
- 哪些安排容易导致后续任务状态下降。

**Review 回顾**  
回答“基于执行记录和节奏信号，下一步要如何调整”。Review 可以发生在每日、每周、每个目标阶段结束后。

Review 需要输出：
- 目标进展评估。
- Routine 坚持情况。
- 任务完成情况。
- 日程安排偏差。
- Rhythm 模式发现。
- 下一周期计划调整建议。

#### 4.2.6 通用执行反馈闭环
所有 Schedule Block 执行后都应该产生 Rhythm Feedback。Agent Adjustment Suggestion 不是反馈本身，而是 Agent 基于执行状态、Rhythm Feedback 和 Review 之后给出的调整建议。

```text
Schedule Block
  -> Execution Record
  -> Rhythm Feedback
  -> Agent Evaluation
  -> Agent Adjustment Suggestion
  -> User Confirmation
  -> Updated Task / Schedule Block
```

**Execution Record** 记录“结果”：完成、未完成、改期、实际耗时。

**Rhythm Feedback** 记录“体验”：顺畅、有阻力、勉强完成、状态很好、状态很差、被打断、没开始，以及可选自然语言补充。

**Agent Evaluation** 读取执行结果和 Rhythm Feedback，判断任务执行过程是否舒适、是否有阻力、是否需要调整。

**Agent Adjustment Suggestion** 给出调整建议，例如拆小任务、换时间、降低门槛、调整优先级、重新编排本周任务。

**User Confirmation** 表示用户确认后，系统才更新任务或日程。AI 初期不自动接管日程。

这个闭环适用于所有目标类型：

```text
健身：
训练完成后反馈“有阻力 / 晚上练完不适合写代码”
-> Agent 下次避免把深度写作安排在训练后

Vibe Coding：
写代码任务反馈“被打断 / 启动困难”
-> Agent 建议把任务拆成更小的 30 分钟子任务，或放到上午

英语：
口语练习反馈“勉强完成 / 有羞耻感”
-> Agent 降低任务门槛，改成先录 2 分钟项目介绍

读书：
阅读任务反馈“状态很差 / 看不进去”
-> Agent 建议换成整理笔记或低阻力输入任务
```

#### 4.2.7 核心关系
```text
Goal 目标
  -> Outcome 结果指标
  -> Project 项目
  -> Task 行动任务
  -> Schedule Block 日程块
  -> Execution Record 执行记录
  -> Rhythm Feedback 节奏反馈
  -> Rhythm Signal 节奏信号
  -> Review 回顾

Goal 目标
  -> Outcome 结果指标
  -> Skill 能力
  -> Routine 长期重复行为
  -> Schedule Block 日程块
  -> Execution Record 执行记录
  -> Rhythm Feedback 节奏反馈
  -> Rhythm Signal 节奏信号
  -> Review 回顾
```

#### 4.2.8 示例：健身目标
```text
Goal：提升身体状态
Outcome：体脂 18-19，有清晰的 4 块腹肌，腿部肌肉明显
Skill：制定肌肥大、增力训练周期和单次训练计划
Routine：每周 3-4 次力量训练
Task：设计本周训练计划
Schedule Block：周二 19:00-20:30 腿部训练
Execution Record：完成，实际 80 分钟，深蹲状态一般
Rhythm Feedback：顺畅；晚上训练启动难度低，但训练后不适合继续高强度脑力任务
Rhythm Signal：晚上适合安排力量训练，但训练后不适合继续高强度脑力任务
Review：下周腿部训练前一天减少熬夜，并避免训练后安排深度写作
```

## 5. 产品能力设计

### 5.1 业务能力层
1. 对话界面：这里可以参考置身钉内的 one的设计理念是“事找人”，可以设计好几张卡片类型，来提醒用户今天此时此刻应该去干什么事情。
2. 目标界面：设置目标，查看 Agent 拆解的任务，完成目标和任务的时间线以及 Agent 对当前状态的评估
3. 日程安排界面：以日历形态查看日程安排
	1. 时间块可以与目标、任务挂钩
	2. 日视图时间轴：默认定位到当前时间；若视口上方仍有已开始且待执行的日程块，顶部显示「↑ 上方还有 N 个未完成日程」，点击滚至离视口最近的一条

### 5.2 AI 能力层
1. Agent编排
2. 记忆系统
3. RAG
4. Tool & Skill
	1. 读取数据
...

#### 5.2.1 MVP Agent 架构
MVP 中，用户不应该直接感知多个不同 Agent。用户主动交互的对象应该是一个统一的 User-facing Agent，由它理解用户意图、读取上下文，并调度底层能力完成不同任务。

可以暂时将这个用户可感知的主 Agent 称为 **Rhythm Companion**。

Rhythm Companion 是用户唯一直接交互的目标推进 Agent。它通过调用目标澄清、任务规划、执行回顾、节奏评估和日程调整等底层能力，帮助用户把目标持续推进到真实生活中。

```text
User
  -> Rhythm Companion
      -> Goal Clarification Capability
      -> Planning Capability
      -> Review Capability
      -> Adjustment Capability
      -> Progress Evaluation Capability
```

#### 5.2.2 User-facing Agent：Rhythm Companion
Rhythm Companion 面向用户提供统一对话与指令入口，负责：

1. 理解用户当前意图。
2. 读取目标、任务、日程、执行记录、Rhythm Feedback、Review 等上下文。
3. 判断应该调用哪一种底层能力。
4. 汇总底层能力的输出，生成用户可理解的建议。
5. 在需要修改目标、任务或日程时，请求用户确认。

用户不需要知道自己正在调用哪个底层能力。用户只需要表达自然语言意图，例如：
- 我想创建一个健身目标。
- 帮我把这个目标拆成这周能执行的任务。
- 我今天没完成训练，帮我看看怎么调整。
- 帮我做一下本周回顾。
- 下周我想优先推进 Vibe Coding，帮我重新编排任务。

#### 5.2.3 底层能力设计
底层能力在产品设计上称为 Capability。具体实现时，它们可以是 prompt chain、function call、workflow、tool，或真正的 specialist agent，不必在 MVP 一开始就全部做成独立 Agent 实例。

**Goal Clarification Capability**  
触发时机：用户创建目标时。  
职责：判断目标信息是否足够，并在拆解前主动向用户追问缺失信息。

需要检查的信息包括：
- 目标描述是否清楚。
- 时间范围是否明确。
- 成功标准是否明确。
- 目标类型：Routine 型、Project 型、Skill 型或混合型。
- 用户每周可投入时间。
- 用户偏好和约束。
- 是否需要日程动态安排。
- 是否需要 AI 主动评估和调整。

**Planning Capability**  
触发时机：目标信息足够后，或用户主动请求规划时。  
职责：拆解目标，生成 Outcome、Project、Milestone、Routine、Task，并为 Task 填充执行指导属性。

Planning Capability 生成的 Task 不应该只是待办标题，而应该包含：
- 任务意图。
- 完成标准。
- 建议步骤。
- 预计耗时。
- 所需精力与专注度。
- Rhythm 匹配条件。

**Review Capability**  
触发时机：定时任务，例如每日回顾、每周回顾。  
职责：读取日程执行情况、Execution Record、Rhythm Feedback、未完成任务、延期任务、Routine 稳定性，并生成回顾总结。

每周 Review 需要整合一周的执行情况，而不是只看当天状态。

**Adjustment Capability**  
触发时机：用户主动唤起，或 Review 后用户选择“帮我调整”。  
职责：根据目标、任务、日程、Execution Record、Rhythm Feedback 和 Review，重新编排任务和日程，并生成 Agent Adjustment Suggestion。

Adjustment Capability 可以建议更新 Task / Schedule Block，但不直接替用户改动正式计划。用户确认后，系统才写入更新。

**Progress Evaluation Capability**  
触发时机：Review、用户主动询问目标进展、Milestone 接近完成时。  
职责：基于执行历程给出目标推进建议，但不直接确认 Outcome / Milestone 达成。

它可以输出：
```text
on_track
slightly_delayed
blocked
needs_adjustment
ready_for_user_review
```

但它不能直接把 Outcome 或 Milestone 标记为完成。它只能建议用户检查，例如：

> 根据本周任务完成情况和反馈，我建议你检查这个 Milestone 是否可以确认完成。

#### 5.2.4 Agent 触发机制
MVP 中 Agent 主要有三类触发方式：

1. 用户输入目标时触发：Goal Clarification Capability -> Planning Capability。
2. 定时触发：Daily Review / Weekly Review -> Review Capability。
3. 用户主动唤起：通过对话框或命令触发 Adjustment Capability、Review Capability 或 Planning Capability。

#### 5.2.5 Agent 读写边界
Agent 可以读取用户能够读取的所有业务数据，也可以生成对用户数据的修改建议。为了保持用户掌控感，写入分为两类：

```text
draft write：Agent 生成草稿 / 建议
confirmed write：用户确认后正式写入
```

Agent 可以帮助用户编排日程，也可以在目标判定场景下给用户建议，但不直接替用户确认目标达成，不直接勾选 Milestone 或 Outcome。

这条边界与前文的系统可观测原则保持一致：AI 可以评估、建议和编排，但最终确认权在用户。
