# V0.4.0 目标执行系统优化

## Goal

让目标页面表达“用户是否真正向目标前进”，而不是表达“系统里创建并完成了多少任务”；同时让里程碑和目标状态能够指导下一步行动。

## Background

- 当前目标卡片旁的进度由任务数量和完成情况计算，计划颗粒度会直接扭曲进度。
- 用户希望考虑展示投入时间，或设计额外的游戏化成就反馈。
- 当前里程碑需要更智能，但具体机制需要结合现有模型从第一性原理拆分。
- “待澄清”与“已确认”目前缺少明确、可感知的行为差异。

## Requirements

### R1. 目标概览反馈

- 盘点当前进度计算、时间记录、执行记录与目标聚合链路。
- 区分计划完成、执行投入、阶段成果和目标达成，不再用单一百分比混合表达。
- 目标列表主反馈固定为“本周真实投入时间 + 当前行动状态”，不展示任务完成比例或其他目标完成百分比。
- 阶段成果和成就作为次级信息；详细证据可进入目标详情下钻查看。

### R1A. 可复用成就事件系统

- 设计版本化成就定义集合，至少包含成就 ID、适用目标类型、标题、说明、图标语义、触发条件、难度层级和解锁证据。
- 成就触发必须来自可审计且能确定归属当前 Goal 的事实：真实投入、执行日期、Routine 执行、带完成记录的 Task、Milestone/Outcome 用户确认等；不得由任务拆分数量、无法关联目标的 Review 或不可解释 AI 分数触发。
- `project / skill / routine / mixed` 使用不同的默认成就组合，同时共享基础成就；新目标按类型自动获得适用集合，不需要复制规则。
- 以当前数据库已有目标和关联数据作为首批成就实例设计样本，并保留旧数据回溯判定能力。
- 目标详情页增加 Steam 风格成就卡片区：图标、标题、公开条件、解锁时间或进展证据、难度层级；不设计隐藏成就。
- 有效事实一旦解锁成就，普通暂停、改期、目标状态变化或后续未达阈值不得撤销；只有源数据被明确更正为错误时才允许审计撤销。
- 视觉仅参考 Steam 成就卡片的信息结构与收藏感，沿用 Rhythm & Routine 的暖白、紫/鼠尾草绿/金/珊瑚色设计语言，不复制品牌图标或深色商店界面。

### R2. 智能里程碑

- 从“阶段成果证明”出发定义里程碑，而非仅作为任务分组或任意日期节点。
- 定义人工创建、Agent 建议、用户确认、执行证据更新、偏离预警和完成判定。
- 保留用户编辑、拒绝和手动维护路径，Agent 不得静默改写用户目标。
- Agent 可在有新证据时自动生成“建议检查”记录，无需 ChangeSet；该记录与 Milestone 生命周期分离。
- 自动建议必须包含证据摘要、生成时间和稳定指纹；用户可确认完成、稍后处理或驳回并填写原因。
- 驳回后进入冷却期；相同证据不得重复建议，只有实质新证据或冷却期结束后才可重新评估。

### R3. 目标状态

- 删除“待澄清/已确认”持久化维度，不新增 confirmed 字段。
- `Goal.status` 只保留 active / paused / completed / archived；新目标默认 active。
- 目标结构不完整时，使用动态、非阻塞 planning hints；Agent 在具体请求中继续推断、查询或追问，但不依赖 readiness 状态。
- 现有 DRAFT 目标迁移为 ACTIVE；兼容既有数据库和 browser-local 数据。
- 采用扩展/收缩迁移：兼容代码先停止产生 DRAFT，再备份与归一化数据，验证后才从 PostgreSQL enum、API 和客户端类型中删除 DRAFT。

### R4. 轻量执行反馈 V2

- 日常反馈只要求回答“这次产生了什么进展”；结果固定为 `achieved / progressed / no_progress`，分别展示为“达成预期 / 有效推进 / 未能推进”。`rescheduled` 是计划动作，不再与完成结果并列呈现。
- 用户可选记录一项专注体验：`deep_focus / steady_focus / under_challenged / overloaded / fragmented`。这组信号分别支持保护深度工作窗口、维持普通节奏、提高挑战、拆小任务和减少注意力切换。
- 完成质量继续保留但改为轻量三级：`satisfying / expected / needs_rework`；补充感受保持可选且在回顾、任务总结和节奏信号中优先于机器标签。
- 实际时长是投入事实而非主观反馈：达成预期默认使用计划时长；有效推进必须记录大约投入分钟；未能推进默认 0。实际起止时间放入折叠详情，不占据主记录路径。
- “阻碍、下一步、舒适度、时间匹配、旧 tags”停止作为 V2 新记录必填/主字段。历史值原样保留并可在兼容详情查看；V2 下游不得把这些保留字段误当作本次新反馈。
- 普通 ScheduleBlock 与 Routine occurrence 使用同一 V2 产品语义。存储使用版本标记和增量 nullable 字段；旧记录保持 V1，不批量改写，不删除，不用推断结果覆盖原值。
- 日/周回顾与 Agent 节奏信号优先读取 V2 outcome/focus/quality/note；旧记录继续通过兼容投影贡献原有顺畅/阻力统计。

## Acceptance Criteria

- [x] 能用现有数据明确计算目标投入时间，且说明统计口径、遗漏与重复边界。
- [x] 目标列表不再把任务完成比例标记为目标进度；替代方案有明确主次层级。
- [x] 每张目标卡片展示本周真实投入时间，并展示一个能指向下一步的行动状态；没有执行数据时显示诚实空状态而非 0% 进度。
- [x] 当前数据库中的每种目标类型均能映射到一组基础成就和类型专属成就。
- [x] 新建相同类型目标时能直接复用成就定义，不新增目标专属代码。
- [x] 每次解锁能追溯触发事实和时间，重复计算不会重复解锁。
- [x] 错误数据更正可以撤销成就并记录 `revokedAt`、`revokeReason` 与原始解锁证据；普通编辑和业务状态变化不能触发撤销。
- [x] 目标详情页能展示已解锁、进行中和未解锁成就，全部条件公开，并在桌面与窄屏保持清晰可操作。
- [x] 里程碑具有可验证成果、目标时间窗、状态和证据来源，并有用户确认边界。
- [x] 给出至少三个真实场景验证里程碑建议：新目标、执行中偏离、阶段成果达成。
- [x] 自动建议不会修改 Milestone 标题、说明、完成标准、目标日期或完成状态。
- [x] 用户确认完成是 Milestone.completed 的唯一入口；暂不处理和驳回不会误标完成。
- [x] 目标生命周期只表达真实业务状态；规划信息缺口作为非阻塞提示或 Agent 运行时判断，不再伪装成目标执行状态。
- [x] 手动创建、Agent 规划和 browser-local 创建均不再产生 DRAFT。
- [x] 迁移前生成可恢复备份和基线摘要；迁移后验证所有实体数量、ID、外键关系和非 status 字段未变化。
- [x] 旧 DRAFT 目标的历史投入、任务、里程碑、Routine、日程、执行、成就回溯结果完整保留。
- [x] 既有目标在迁移或缺失新字段时仍可查看和手动管理。
- [x] 新反馈弹窗的默认路径不超过四组输入，改期和实际时间详情不与执行结果混为一组。
- [x] 有效推进必须有真实投入分钟；未能推进不会因计划时长回退而产生虚假投入。
- [x] 普通日程和 Routine 都能保存 V2 结果、专注体验、质量和补充感受。
- [x] 所有 V1 执行记录及其阻碍、下一步、舒适度、时间匹配和 tags 在升级前后字段值不变，仍能查看并参与兼容分析。
- [x] 回顾与节奏信号能分别识别深度投入、挑战偏低、挑战偏高和注意力破碎；数据不足时不推断。

## Out of Scope

- 不在首版引入全局排行榜、社交竞争或可兑换奖励。
- 不用不可解释的单一 AI 分数宣称目标完成程度。

## Confirmed Decisions

- 2026-07-20：目标列表使用“本周真实投入时间 + 当前行动状态”作为主反馈；成果/成就作为次级信息；不再展示目标完成百分比。
- 2026-07-20：加入一组成就事件集合；按目标类型选择不同成就组合并支持后续目标复用；首批设计参考当前数据库已有目标与属性；在目标页采用类似 Steam 成就卡片的信息结构。
- 2026-07-20：已解锁成就永久保留；普通状态变化不撤销；仅错误源数据更正可审计撤销。
- 2026-07-20：不设计隐藏成就；所有条件和阈值公开。
- 2026-07-20：硬删除 Goal readiness（DRAFT）语义，不新增 confirmed；最终 schema 只保留生命周期状态；迁移必须保证除 DRAFT→ACTIVE 外当前数据零变化。
- 2026-07-20：Agent 可自动生成里程碑检查建议，无需 ChangeSet；建议不能改定义或确认完成；用户拥有确认、稍后、驳回控制，系统负责证据去重与冷却。
- 2026-07-20：人工快速记录 Milestone 时完成标准可选；只有具备公开、可机器判定完成标准的 Milestone 才参与智能建议。
- 2026-07-21：执行反馈 V2 拆分“进展结果、专注体验、完成质量、补充感受”；改期作为独立计划动作；旧低频字段只做兼容保留。
