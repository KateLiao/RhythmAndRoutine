# V0.4.0 优化计划

## Goal

以“更真实地反映目标执行”和“让 Agent 更准确、更可测、更快”为核心，形成 V0.4.0 可直接进入开发的版本规划，并保证两个方向能够独立实施、独立验收，最终完成跨模块集成验证。

## Background

- 当前目标列表使用任务设计数与完成数计算进度，容易把“计划得多”误解为“目标推进得好”。
- 当前里程碑需要从静态任务分组升级为能够帮助用户判断阶段成果、风险与下一步的执行机制。
- 目标的“待澄清”与“已确认”在用户认知、可执行条件和系统行为上的边界不清楚。
- Agent 已具备多个能力与执行追踪基础，但缺少统一能力清单、可重复的 QA 评测反馈闭环，以及针对意图识别、任务拆解和工具并行的专项优化。

## Requirements

### R1. 目标执行系统

- 重新定义目标列表的主反馈指标，停止把任务完成比例直接等同于目标进展。
- 目标列表主反馈采用“本周真实投入时间 + 当前行动状态”；成果与成就只作为次级信息，不展示目标完成百分比。
- 建立可跨目标复用、按目标类型适配的成就事件集合；首批规则以当前数据库中的真实目标及其属性为设计样本。
- 目标页面加入 Steam 风格的成就卡片区域，明确区分已解锁、进行中和未解锁成就，同时保持产品现有视觉语言；所有成就条件公开，不设计隐藏成就。
- 成就由有效事实解锁后永久保留；普通暂停、改期或后续状态变化不得撤销。只有修正错误源数据时可审计撤销。
- 从第一性原理定义里程碑职责、生成方式、状态变化与用户控制边界。
- Agent 可基于真实执行与回顾证据自动生成“建议检查里程碑”，无需 ChangeSet；建议不得修改里程碑定义或确认完成，最终完成仍由用户操作。
- 删除 Goal 的“待澄清/已确认”持久化维度，不新增 confirmed 字段；`Goal.status` 只表达 active / paused / completed / archived 生命周期。
- 新目标默认 active；现有 DRAFT 目标安全迁移为 ACTIVE。迁移除 status 归一化外不得改变任何当前业务数据或关系。

### R2. Agent 质量与效率

- 梳理当前 Agent 能力、边界、输入输出、工具依赖和失败模式。
- 建立覆盖各能力的可重复 QA 评测、评分、回归与反馈闭环。
- 提升意图识别与任务拆解准确率，并对不确定性和澄清行为建立可测标准。
- 提升执行速度；无数据依赖且无写冲突的工具调用应支持安全并行。

### R3. 版本边界

- 子任务一：`07-20-v0-4-0-goal-execution`，负责目标执行系统。
- 子任务二：`07-20-v0-4-0-agent-quality-efficiency`，负责 Agent 能力、评测与效率。
- 本轮仅完成规划、研究、技术设计和实施顺序，不启动产品代码实现。

## Acceptance Criteria

- [x] 两个子任务均形成经过代码事实核查的 `prd.md`、`design.md` 与 `implement.md`。
- [x] 每项需求都有用户可见结果、数据/接口/交互影响、边界条件和可执行验收场景。
- [x] 目标执行指标、里程碑机制和目标状态形成一致的产品模型，不再混用“计划完成率”与“目标真实推进”。
- [x] 目标列表展示本周真实投入及行动状态，不展示由任务数、里程碑数或 AI 评分推导的目标完成百分比。
- [x] 成就规则按目标类型选择，复用同一规则定义与触发基础，不为每个目标硬编码一套逻辑。
- [x] 首批成就覆盖当前数据库中已有目标类型，并能以真实执行、回顾或用户确认事件确定性判定。
- [x] 目标页的成就 UX 参考 Steam 卡片的信息结构，但不复制 Steam 品牌素材；全部解锁条件公开，未解锁状态不会制造虚假目标进度。
- [x] 普通业务状态变化不会让已解锁成就消失；错误数据更正触发的撤销保留撤销时间、原因和原解锁证据。
- [x] Agent 能力清单与 QA 用例一一映射，并定义准确率、成功率、时延和回归门槛。
- [x] 并行执行有明确的依赖判定、写冲突与失败处理边界，而不是无条件并发。
- [x] 给出 V0.4.0 分阶段实施顺序、依赖关系、风险和回滚点。
- [x] 已经存在有效日程或真实投入的目标不会再因无行为意义的 readiness 标签显示为“待澄清”。
- [x] Goal 状态迁移前后 Goal、Outcome、Milestone、Task、Routine、ScheduleBlock、ExecutionRecord 的数量、ID、关联和业务字段保持一致；唯一允许的业务字段变化是旧 DRAFT → ACTIVE。
- [x] 相同里程碑与相同证据不会重复产生提醒；用户驳回或稍后处理后遵循冷却期，出现实质新证据才可提前重新建议。

## Out of Scope

- 本轮不修改产品代码、不迁移数据库、不发布版本。
- 不把所有确定性 LLM 工作流强制改造成 Agent Loop。
- 不在尚未定义产品价值前直接引入完整游戏化等级、积分或社交排行系统。

## Confirmed Decisions

- 2026-07-20：目标列表主反馈使用“本周真实投入时间 + 当前行动状态”；成果/成就作为次级信息；移除目标完成百分比。
- 2026-07-20：V0.4.0 纳入一组成就事件集合；按目标类型使用不同成就组合，可被后续目标复用；首批设计以当前数据库真实目标为参考；在目标页使用类似 Steam 信息结构的成就卡片。
- 2026-07-20：成就解锁后作为历史收藏永久保留；普通暂停、改期或状态变化不撤销；仅错误数据更正可审计撤销。
- 2026-07-20：不设计隐藏成就；所有成就条件和阈值均向用户公开。
- 2026-07-20：删除 Goal readiness（DRAFT/已确认）语义，不新增 confirmed 字段；Goal 只保留生命周期状态；采用零损失迁移，现有 DRAFT 仅归一化为 ACTIVE，其他数据不得变化。
- 2026-07-20：Agent 可无需 ChangeSet 自动生成里程碑检查建议；不得修改定义或确认完成；建议需支持证据、去重、驳回和冷却。
- 2026-07-20：人工快速记录里程碑时完成标准可选；只有具备公开、可机器判定标准的里程碑才参与智能建议。
- 2026-07-20：Agent QA 首发为 Router 120、Planner 30、Runtime 30、Performance 10；PR 跑确定性集，发布前跑真实模型抽样，失败样本人工复核。
