# 优化 Agent 对话窗口与上下文

> 父任务：`07-09-v03-release`  
> 需求文档：`docs/v0.3.0 版本需求/特性优化-Agent 对话窗口与上下文.md`  
> 状态：in_progress · 核心实现已落地，待手动验收  
> 相关：`docs/v0.2.0 版本需求/特性优化-AgentLoop优化需求描述.md`

## Goal

让用户在对话窗中始终能回答三件事：

1. 小律**现在在做什么**？
2. 它基于什么信息、**调用了什么**（含如何调用）？
3. 清空上下文 / 新建对话后，**哪些继续存在、哪些停止或消失**？

成功标准（可验证，不用「Agent 感」验收）：

- 能看到当前真实动作（status = 活动步骤投影）
- 能展开查看该动作使用的输入（脱敏/限长）
- 能看到动作结果、失败、取消或下一步
- UI 状态不与服务端 Run / ChangeSet 状态矛盾

## Background

v0.3 清单第 3 项四条主线：窗口、真实过程披露、历史上下文、Loop 呈现。执行质量可接受；本任务不重做 Runtime，只调整真实事件的披露与 Session/上下文生命周期。

**继承且不削弱（v0.2）**：上下文优先推断再追问、有界失败恢复、ChangeSet 审批、结构化退出原因与既有 Loop 事件。本任务可减少对用户无信息量的重复展示，但不得删掉可调试的真实事件数据源。

## Decisions

| 决策 | 结论 |
|------|------|
| 范围 | A–D 都做；C = 呈现层，不重做执行质量 |
| 动态感来源 | 真实 SSE；无 thinking token；无 LLM 旁白 |
| 工具披露 | 中文名 + 可读入参摘要；展开=脱敏限长参数；不含 HTTP 实现细节 |
| 窗口 | 默认加宽 + 记住展开；展开 >700；丝滑动效；内容独立 max-width |
| 过程默认态 | **执行中默认展开；最终回复出现后自动收为一行；完整参数始终手动展开**；用户手动展开后本轮不自动收起 |
| 视觉语言 | 延续现有暖白、紫/鼠尾草绿/金/珊瑚色；过程使用产品化「节奏轨」，不引入通用蓝紫霓虹 AI 风格 |
| ChangeSet UX | 草案是独立确认层，不混成普通气泡；继续对话时保持可见，滚离后在输入区上方显示待确认提醒 |
| 新建交互 | 无 Run/草案时直接新建；有 Run 或草案时先用明确后果文案确认，不使用笼统二次确认 |
| Session | 仅当前 Session 存 localStorage；新建后**删除**旧 Session 本地消息；无列表/不可跳回 |
| 装载 | 滑动窗原文 + 异步 best-effort 摘要；不阻塞发送/SSE |
| 清空 | 同 Session；边界绑定 messageId/revision，不只靠时间戳；不拒草案、不取消 Run |
| 新建对话 | **终止当前任务生命周期**：停 SSE、取消 Run、拒绝本 Session 关联 pending ChangeSet、换 Session；**必须有用户可感知提示** |
| ChangeSet 归属 | 只处理与**本 Session 已记录的 runId**关联的 pending ChangeSet；不扫全用户 pending |
| 落盘 | 不 merge AgentRun 进面板 |
| 切目标 | 仅 `selectedGoalId` 变化 → 清空上下文；普通换页不清空 |
| 摘要 | best-effort 浏览器触发；关页可不完成；带 sessionId+revision 防污染 |
| P0-1 新建×Run | **已确认：是，取消正在执行的 Run**，并给出相应提示让用户感知 |

## Confirmed Facts（准确表述）

- 面板默认 400 / 展开 700；有展开按钮
- `tool-labels` 可摘要工具**结果**；`tool_started` 尚无稳定 toolCallId / 入参摘要；前端用「同名最后一个 running」匹配 completed（并发同工具会错配）
- 对话消息仅 localStorage；AgentRun 在 DB；面板会 merge Run（本任务去掉）
- **ContextBuilder 仅预留** `conversationSummary` 入口；客户端存储、请求协议、生成、并发控制与追溯**均未实现**（不是「字段已有未接线」的小活）
- `streamChatWithAgent` 支持 AbortSignal，面板当前未传入；服务端已有 cancel Run（并 reject 关联 ChangeSet）
- `listPendingChangeSets` 返回用户全部 pending，面板取第一份——无 Session 归属

## Requirements

### R1 对话窗口
- 桌面默认约 520px，展开约 880px（均受视口边距限制）；记住展开状态
- 右侧固定为视觉锚点，面板向左舒展；动效约 220–280ms、可打断，展开不改变滚动位置或输入焦点
- 助手正文最大阅读宽度约 640–680px；用户气泡约为内容列 75%；过程区 / ChangeSet 可略宽但不铺满面板
- Header 将「新对话」与「清空上下文」作为同级操作直接展示；两者共享按钮层级，以不同图标/色彩区分；展开/关闭保留图标按钮
- 移动端全屏

### R2 动态处理过程（事件→UI 映射）

展示模型（可验收）：

| 步骤类型 | 规则 |
|----------|------|
| 理解/准备 | 同一 Run 最多 1 条 |
| 工具调用 | 每次真实调用 1 张卡，按 `toolCallId` 关联 started/completed/failed/cancelled |
| 失败恢复 | 仅真实失败时出现 |
| 等待确认 | 最多 1 条 |
| 最终输出 | **不再**单独一张与回复正文重复的卡 |
| status | 仅当前活动步骤投影，不另造叙事 |

工具卡：

- 中文展示名 + 一行入参摘要；展开=脱敏/限长参数
- 分三层披露：行动/结果摘要 → 格式化 key-value 技术详情 → 用户主动点击后才显示原始 JSON
- 结果摘要 + 已有 detail
- 断流/取消/异常：running 必须变为 cancelled/failed，禁止永久转圈
- 敏感字段脱敏；长数组/文本截断；最大展开尺寸；不可序列化则降级文案
- 默认不展示内部工具注册名；内部名仅在技术详情中以弱化等宽文字出现

过程交互：

- 处理过程位于当前助手回合内，采用「节奏轨」串联真实步骤；当前=紫色呼吸节点、完成=鼠尾草绿、确认=金色、失败=珊瑚色，同时必须有图标/文字，不能只靠颜色
- 执行中默认展开；当前步骤始终可见
- 最终回复开始出现后，若用户未手动展开详情，过程平滑收成「已完成 N 步 · 摘要 · 查看过程」一行
- 用户本轮手动展开任一步骤后，本轮完成时不自动收起，尊重用户操作
- 输入框上方可显示粘性当前状态；它只投影当前步骤，点击可滚动到活动回合，不与过程卡重复叙述

### R3 Session / Run / ChangeSet / 摘要

#### Session 生命周期

| 事件 | 行为 |
|------|------|
| 首次打开 / 无有效存储 | 创建 Session；欢迎消息属于该 Session（可不装载给模型） |
| 刷新 | 恢复**当前** Session（消息、边界、summary、关联 runIds） |
| 新建对话 | 见下方状态机；本地**删除**旧 Session 数据 |
| 清空上下文 | 设 `contextBoundaryMessageId` + bump `revision`；清 summary；消息保留 |
| 结束 | 仅因「新建」替换；无单独「结束」UI |
| localStorage 损坏/超限 | 新建空 Session；不崩 |
| 多标签 | 后写覆盖；以 `revision` 丢弃过期异步结果（不做跨 tab 实时同步） |
| v1→v2 迁移 | 一次性：原消息迁入唯一 active Session，保留 cleared 边界语义 |

Session 至少保存：`sessionId`、`revision`、`messages`、`contextBoundaryMessageId`、`summary`、`summarizedThroughMessageId`、本 Session `runIds[]` / 当前 `activeRunId`、关联 `pendingChangeSetIds[]`。

#### 新建对话状态机（P0）

```text
用户点击「新建对话」（Run 进行中也可点）
  → bump UI generation / 停止接收旧 SSE（AbortController）
  → 若有 activeRunId：POST cancel（取消 Run + 拒该 Run 关联 pending CS）
  → 另：对本 Session 记录的其余 pendingChangeSetIds 执行 reject（兜底）
  → 使旧摘要任务失效（revision 不匹配则丢弃写回）
  → 删除旧 Session 本地数据，创建新 Session + 欢迎态
  → 【用户感知提示】按结果展示（见下表），不得静默完成
  → 若 cancel/reject 未全部成功：进入新 Session，但在非对话历史的持久提示区
       显示「仍有未处理草案/未取消任务」，提供重试清理
  → 任何旧 SSE / 旧摘要回调：sessionId 或 revision 不匹配则丢弃，不得写入新 Session
```

#### 新建对话 · 用户感知提示（必做）

| 情况 | 用户应感知到 |
|------|----------------|
| 新建时有进行中的 Run | 明确提示：已停止当前处理 / 已取消本次任务（短 toast 或面板内状态条均可，文案需可读） |
| 新建时有本 Session 待确认草案被拒绝 | 提示：已放弃待确认的变更草案 |
| 仅空闲新建（无 Run、无草案） | 轻量提示即可：已开始新对话（可选，避免打扰） |
| 清理部分失败 | **持久**警告（非一闪而过）：仍有未取消的任务或未处理草案 + 重试入口 |
| 过程卡 / status | 若用户仍短暂看到旧步骤，running 应变为 cancelled，与「已停止」一致 |

前端确认规则：

- 无进行中 Run、无待确认草案：点击「新对话」直接进入欢迎态，不弹框
- 有进行中 Run：确认文案必须说明「停止当前处理并开始新对话」，主操作为「停止并新建」
- 有待确认草案：确认文案必须说明「草案会被放弃，正式计划不会改变」，主操作为「放弃草案并新建」
- 同时存在 Run + 草案：合并说明两项后果，只确认一次
- 清理失败后仍进入新 Session，但持久警告必须保留「重试」入口

#### ChangeSet「当前」定义

- **只**拒绝与本 Session 已关联 `runId` / `pendingChangeSetIds` 的 pending 草案
- 不拒绝其他 Session、其他目标、列表中「碰巧排第一」的无关草案
- 因此 Session 必须记录 runId↔changeSetId；仅靠随机 sessionId 不够

#### 清空上下文 vs 新建

| | 清空上下文 | 新建对话 |
|--|-----------|----------|
| Session | 同一 | 新（删旧本地） |
| 消息 UI | 保留，标界外 | 欢迎态 |
| Run | 不取消 | **取消** |
| ChangeSet | 不动 | 拒本 Session 关联 pending |
| 摘要 | 清空并失效进行中任务 | 删除 |

清空上下文的可见反馈：

- 在准确边界插入持久分隔线：「上方内容已移出上下文，小律接下来不会参考上面的对话」
- 旧消息保持正常可读，不用大面积低透明度表达边界
- 在用户发送下一条消息前提供「撤销」；发送后边界生效且撤销入口消失
- 自动切目标也插入同类系统分隔线，并写明新目标名称；不能只用短暂 toast

#### 切目标

- 触发条件：**仅** `selectedGoalId` 变化（含 A→B、A→未选、未选→A）；`view` 变化单独不清空
- 行为：等同清空上下文
- Run 进行中切目标：先按「终止当前 Run」（同新建中的 cancel 路径）再清空上下文，**不**新建 Session；迟到事件丢弃

#### 异步摘要契约

| 项 | 约定 |
|----|------|
| 触发 | 回合结束后，若界内消息轮次 > N（建议 6）则浏览器 fire-and-forget |
| 内容 | 仅用户/助手正文；不含工具原始 JSON / processSteps |
| 截止 | `summarizedThroughMessageId`；增量叠 priorSummary |
| 进行中又发消息 | 旧任务返回时 revision 不匹配则丢弃；可再触发 |
| 清空/新建后 | 旧摘要禁止落盘 |
| 乱序 | 只接受匹配 `sessionId + revision` 的响应 |
| 长度 | 上限（如 1500–2000 字），超出截断 |
| 规则降级 | 保留：最近界外用户句要点 + 助手结论首句；丢寒暄；可验收 |
| 关闭页面 | 允许不完成（best-effort） |
| 可观测 | Run contextManifest 或等价：`summaryUsed`、`summaryChars`、`summarizedThroughMessageId`、`summaryRevision`（可不存正文） |

### R4 呈现层
- 不引入外部框架；不绕过 ChangeSet 写正式数据
- 不做每轮结构化 Loop Decision LLM
- 摘要/取消均不阻塞「进入新 Session」的主路径（清理可异步重试）

### R5 对话、草案与输入区交互

#### ChangeSet 确认层

- ChangeSet 是独立于助手气泡的确认层：标题、原因、风险/数量、可选择操作、主确认按钮、放弃按钮
- 用户继续讨论时草案不得从 UI 消失；滚动离开草案卡后，输入框上方显示「有一份变更草案等待确认 · 查看」
- 应用成功后卡片就地变为只读「已应用 N 项」；拒绝后就地收为「已放弃这份草案」，不再追加重复助手气泡
- 输入区辅助文案说明「继续讨论不会应用草案」；正式写入仍只由显式确认触发

#### Composer 执行态

- Run 执行中输入框仍可编辑并保留草稿；本版不做消息排队
- 发送按钮切换为明确的「停止」控制；停止后恢复发送
- 面板展开/还原、停止 Run、新建确认框开关均不得丢失输入草稿

#### 欢迎态

- 欢迎态保持轻量，只显示 2–3 个与当前页面/目标相关的快捷入口
- 不加入历史列表、模式切换、模型选择或大型功能宫格

### R6 草案视觉与日程规划约束

#### 对话与草案视觉

- 用户消息必须在消息内容列右对齐；展开窗口后也不得因外层 wrapper 改变而退回左侧
- 待确认 ChangeSet 使用独立“决策卡片”：方案摘要、变更数量、选择状态、操作列表和审批操作区形成清晰层级
- 应用/放弃后的 ChangeSet 收束为轻量处理回执，不得继续表现成大面积输入框或可编辑控件
- 移动端审批按钮保持至少 44px，允许换行但不得横向溢出

#### Agent 日程规划

- 新增或移动 `schedule` / `personal_schedule` 前必须读取覆盖候选时段的现有日程，并按半开区间 `[startsAt, endsAt)` 检查重叠
- 默认避开已有未取消日程；用户明确接受冲突时才可保留，并在草案理由中说明
- 在用户没有自定义偏好时，将本地时间 11:30–13:30、18:00–19:30 作为应尽量避开的常规饭点；这是软约束，不是禁止用户自行指定
- 新增 `read_similar_schedule_history` 只读工具：仅当用户明确要求“照往常/按习惯/和以前一样”时使用，返回相似日程的典型开始时间、典型时长、常见时段与历史样本
- 历史习惯只用于候选排序，不证明本次时段空闲；调用历史工具后仍必须重新检查本次日程窗口
- Runtime 必须阻止未完成当前窗口检查的一次性日程 ChangeSet，不能只依赖模型遵守提示词
- 相似历史查询必须先由独立 planner 根据用户原始请求生成三层候选：完整活动语义（exact）→核心对象（related）→活动类别（broad）；只有上一层零结果时才能进入下一层
- “习惯”默认只使用 `COMPLETED` 历史；`PLANNED` 不能作为已形成习惯的证据，标题中偶然包含宽泛关键词的其他活动不能污染精确层
- 日程窗口工具必须返回用户时区下的本地起止时间、有效日程、合并忙碌区间和可用区间；窗口外 Routine、`CANCELLED`、`RESCHEDULED` 旧记录不进入 Agent 决策数据
- 任何具体时间建议或一次性日程 ChangeSet 前必须调用 `validate_schedule_candidates`；候选冲突、候选未覆盖或校验后又修改时间时不得结束或创建草案

## Out of Scope

- 历史 Session 列表 / 跳回 / 静默保留不可访问的旧 Session
- Conversation DB 表；关页后必须完成的服务端摘要任务
- thinking token / 每步 LLM 旁白
- Runtime 工具选型大改；固定 workflow 套用 Agent Loop

## Acceptance Criteria

1. 窗口：默认/展开尺寸符合 R1；展开偏好可恢复；内容 max-width；展开不改变滚动位置、输入焦点或草稿
2. 工具：同 Run 多次同名工具显示为多张卡；均有 toolCallId 对齐；行动摘要、格式化参数、原始 JSON 三层披露；取消/失败无永久 running
3. status 仅为活动步骤投影；无「已完成仍正在…」；点击可定位活动回合
4. 过程模型符合 R2：执行中默认展开；最终回复出现后自动收为一行；用户手动展开后不自动收；无重复最终卡
5. 新建（含 Run 中）：旧 SSE 不再写入；Run 被取消；用户看到「已停止/已取消」类提示；本 Session 关联草案被拒时有「已放弃草案」提示；失败则持久警告+重试；新 Session 欢迎态；旧消息本地删除
6. 清空：消息保留+边界；草案与 Run 不动；摘要失效
7. 切目标：仅 goalId 变化清空；换页不清空
8. 长对话：后续请求可验证 summaryUsed（manifest/调试）；摘要不阻塞 SSE 首包
9. 冷启动不 merge 历史 AgentRun 进对话
10. 拒绝失败路径：新 Session 可建立，但未拒成功的草案**不能**仅从 UI 静默消失
11. ChangeSet：继续对话和滚动时草案不消失；确认/拒绝后卡片原位进入只读终态，不追加重复回复
12. 新建：空闲时直接新建；有 Run/草案时出现与实际后果一致的确认文案，且一次确认完成
13. 清空/切目标：消息流中有持久上下文边界；清空后在下一次发送前可撤销；旧消息仍清晰可读
14. Composer：Run 中可编辑但不排队；发送键变为停止；停止、展开、新建确认均不丢输入草稿
15. 无障碍/移动端：375px 和移动端全屏无横向页面滚动；交互目标≥44px；键盘可完成展开、停止、新建、确认/拒绝；reduced-motion 下状态仍清晰
16. 用户气泡：默认与展开窗口均在内容列右对齐；助手消息、过程区与 ChangeSet 保持左对齐
17. ChangeSet：待确认态为决策卡，终态为轻量回执；桌面和 375px 下无输入框误导或横向溢出
18. 规划冲突：创建/移动一次性日程前必须成功调用 `read_schedule_window`；未调用时 Runtime 返回可重试错误且不创建草案
19. 习惯安排：只在明确习惯意图下使用历史工具；工具返回聚合时段；调用后必须再次查当前窗口；默认尽量避开常规饭点
20. 过程状态：展开态按真实事件顺序线性追加；同一 toolCallId 完成时原位更新，下一工具追加到底部；逐轮 verification 不进入主时间线；无工具结束不得显示为“验证工具结果”
21. 精确习惯：对于“按习惯阅读《原则》”，先查询“阅读《原则》/原则阅读”；精确层有结果时不得继续用“阅读”宽泛查询，返回样本仅包含已完成的精确活动
22. 查询降级：exact 零结果才查 related，related 零结果才查 broad；工具结果披露命中层级和每层样本数，宽泛命中不得表述成精确活动习惯
23. 忙闲事实：timezone-less 工具入参按用户时区解析；工具响应直接包含本地时间与忙闲区间，并过滤窗口外 Routine 和无效旧记录
