# 修复回顾节奏遗留问题

## Goal

修复 `07-10-review-rhythm-evaluation` 上线后暴露的导航状态、回顾页信息架构与任务完成语义问题，让导航提示由真实待办驱动、日/周回顾各自表达正确，并纠正被错误完成的存量任务。

## Background

- 来源任务：`.trellis/tasks/archive/2026-07/07-10-review-rhythm-evaluation/`
- 来源需求：`docs/v0.3.0 版本需求/新增特性-回顾节奏评估.md`
- 当前用户可见问题：
  1. 左侧导航「今天」固定显示假 badge `4`，「回顾」固定显示假标签「新」。
  2. 日回顾与周回顾主标题都固定显示「这一周，节奏正在成形」。
  3. 日/周切换混在 header hero 内，周回顾沿用日回顾布局，无法清晰承载任务、Routine、目标检查与确认操作。
  4. 关联任务仍可能因同名时间块完成而被误判完成；现有误完成数据需要恢复为等待完成。

### Confirmed Facts

- `src/components/product-shell.tsx` 中「今天」的 `hint="4"` 与「回顾」的 `hint="新"` 均为硬编码。
- 顶层 `viewMeta.review.title` 固定为「这一周，节奏正在成形」；`ReviewView` 虽已按 Tab 读取各自 `summary`，但无法改变顶层页头，形成双标题与错误周语境。
- `ProductShell` 已持有 `schedule`、`reviews`、`goals`、用户时区，足以派生真实导航提示，无需新 API。
- 当前服务端 Task↔ScheduleBlock 关联只使用 `taskId` / `ScheduleBlockTask`，未发现标题匹配路径；同名现象是旧自动聚合留下的数据表现，不是当前 title join。
- `aggregateTaskStatus` 当前已禁止把 Task 聚合为 `completed`，但会保护既有终态，因此历史误完成数据不会自动恢复。
- `PATCH /api/tasks/:id` 的 `updateTask` 仍允许 `status=completed`，可绕过 `completeTaskWithSummary` 且不写 `completionRecord`。
- 当前数据库共有 7 个 `completed` Task，全部缺少 `completionRecord`，均符合历史自动聚合或旁路完成特征；当前没有带用户确认完成记录的 `completed` Task。

## Requirements

### R1. 导航提示真实化

- 「今天」badge 必须统计用户时区当天仍需处理的日程块：`planned` / `in_progress`，包括已经过期但尚未记录执行结果的块；完成、已记录未完成、改期、取消后从计数中移除；数量为 0 时隐藏。
- 「回顾」提示必须来自真实待处理回顾状态：至少存在一份 `awaiting_confirmation` 回顾时显示「新」，全部确认后隐藏；用户仅进入页面不应清除提示。
- 提示需要具备可解释、可稳定复算的口径，并与页面实际可操作内容一致。

### R2. 日/周回顾标题正确

- 日回顾主标题必须使用当前日回顾自身的 `summary` 或日回顾专用空态/生成态文案。
- 周回顾主标题必须使用当前周回顾自身的 `summary` 或周回顾专用空态/生成态文案。
- 禁止以固定周回顾标语覆盖日回顾内容。

### R3. 回顾页信息架构调整

- 将「日回顾 / 周回顾」切换移出 header hero，作为页面级分段导航。
- 日回顾保持轻量收工复盘：周期、状态、摘要、关键发现、轻量建议与主要操作。
- 周回顾采用更高信息密度的独立布局，清晰承载周指标、关键发现、任务进展、Routine 稳定性、目标检查、下周建议与「只由你确认」操作。
- 日/周继续共用可复用的视觉与内容组件，但不能强行使用完全相同的信息层级。
- 桌面与窄屏均保持可读、可操作，不逐条复刻完整执行明细。

### R4. 任务完成语义与存量数据修复

- ScheduleBlock 只能通过稳定关联标识（如 `taskId`）影响关联任务的推进状态；标题相同不得建立或推断任务关联。
- ScheduleBlock 完成不得自动把 Task 标记为 `completed`；Task 完成只允许走用户确认完成路径。
- `PATCH /api/tasks/:id` 不得接受 `status=completed`；完成必须走 `/api/tasks/:id/complete` 并写入 `completionRecord`。
- 对当前数据库中由错误自动聚合产生的已完成任务，恢复为等待用户完成的非终态；不得回滚有明确用户确认完成证据的任务。
- 数据修复必须可审计、可重复安全执行，并在执行前后输出影响数量。
- 周回顾任务进展汇总必须同时覆盖主 `taskId` 与 `ScheduleBlockTask` 多任务关联，避免非主关联任务遗漏。

### R5. 文档与回归保护

- 同步更新 `docs/v0.3.0 版本需求/新增特性-回顾节奏评估.md`，记录最终导航口径、日/周布局与数据修复规则。
- 为导航派生状态、日/周标题与任务聚合逻辑补充适当回归测试。

### R6. 定时生成漏跑恢复

- Vercel Cron 仍作为部署环境的主动定时触发，但生成判定不得只接受与设置时间完全相等的单分钟；Cron 延迟或漏跑后再次触发时，必须能识别并补生成最近已到期的日/周回顾。
- 数据库模式下，应用启动读取回顾时必须执行同一套到期同步，保证本地 `npm run dev/start` 没有外部 Cron 时也能在用户再次打开应用后恢复最新已到期回顾。
- 到期同步必须复用 `generateReview` 与现有周期幂等键；已有非失败记录不得重复生成，失败记录允许重试。
- 自动恢复只补最近已到期的日回顾与周回顾，不在一次页面加载中批量调用多日 LLM；更早历史缺口不自动回填。

## Acceptance Criteria

- [x] 日/周切换位于 hero 外；周回顾在桌面和窄屏下均能清晰展示周指标、任务/Routine/目标建议和确认区。
- [x] 两个同名任务/时间块不会串联；完成时间块后关联 Task 仍保持非 `completed`，仅用户确认后完成。

- [x] 存量误完成任务已恢复为待完成状态；有明确用户完成记录的任务保持 `completed`；修复结果有数量记录。
- [x] 周回顾任务进展包含仅通过 `ScheduleBlockTask` 关联的非主任务。
- [x] 修复前 7 月 12、13 日均无日回顾记录；已对两个缺失周期完成一次性幂等补生成，状态均为 `awaiting_confirmation`。
- [x] Cron 在设置时间之后才触发时仍能生成最近到期回顾；本地无 Cron 时，读取回顾也会执行相同的到期同步，重复同步不新增记录。
- [x] 类型检查、lint、构建及相关测试通过。
- [x] 需求文档与最终实现一致。
