# V0.4.1 Agent 提案续接与增量调整

## Goal

让 Agent 把同一对话中的后续反馈识别为“修订上一份提案”，复用结构化提案和仍然有效的证据，只重新读取与校验发生变化的部分；在不降低日程冲突、ChangeSet 确认和审计安全性的前提下，显著减少模型轮数、重复工具调用与 Token。

## Background

真实场景中，用户第一轮要求安排四项当日日程，第二轮只说“第一个日程可以从 10.15 开始，其他没问题”。当前系统把第二轮当成全新的 `adjustment` Run：重新读取全天日程、重新校验四个候选，并重新生成四项 ChangeSet。

已核对的基线：

- 第一轮：6 个 Loop、6 次工具调用、23,843 Token。
- 第二轮局部修改：4 个 Loop、3 次工具调用、14,326 Token，约为第一轮的 60%。
- 第一轮最终只输出自然语言方案，没有创建可供续接的结构化 ChangeSet；第二轮只能从对话文字重建四项安排。
- 当前最近对话和摘要只携带自然语言；`ToolEvidenceLedger`、成功工具序列与候选校验证据均以单个 Run 为边界。
- 当前创建型日程 ChangeSet 在应用时没有覆盖日程窗口的版本/指纹校验，不能用“对话阶段反复检查”代替真正写入边界的最终安全校验。

## Adjustment Intent Model

V0.4.1 将 `adjustment` 细分为以下子意图；仍共用既有 capability 与 ChangeSet 确认边界，不新增通用子 Agent。

### A. `itinerary_create`

用户安排一日或多日行程，例如“帮我安排今天”“把这三天排一下”。

- 基本保持当前完整流程：识别活动与约束、读取覆盖全部候选的日程窗口、按需读取目标关系、生成候选、校验全部最终候选、创建待确认 ChangeSet。
- 对明确要求“安排/加入日历”的请求，校验完成后必须创建真实待确认 ChangeSet；不能只输出文字并询问是否确认。
- 一日与多日只在窗口范围和候选数量上不同，安全规则一致。

### B. `proposal_patch`

用户调整上一份一日或多日提案，但不改变全部活动，例如补充约束、修改某一项内容、删除或新增一项。

- 服务端按当前 Conversation、父 Run 和有效待确认 ChangeSet 定位上一份结构化提案。
- 读取对应 ChangeSet、稳定 operation ID、提案涉及的行程窗口和必要的新鲜度信息；不得依靠助手自然语言重建整份提案。
- 只修改被用户点名的 operation；未变 operation 的结构化字段、关联目标与已验证事实原样继承。
- 无法唯一定位“第一个/银行那个/下午那项”时，只追问目标项，不重跑完整规划。

### C. `proposal_reorder`

用户调整上一份提案中活动的执行顺序。

- 读取对应 ChangeSet 与受影响的连续行程片段。
- 用户只指定顺序、不指定新时间时，必须调用当前选定的大模型，让模型基于活动语义、时长、优先级、固定约束、可用窗口、饭点和已有节奏信息推理新的合理时间，而不是由工具机械交换两个时间字段。
- 模型保留活动时长、关联目标、任务和非时间字段，只能为受排序影响的 operation 生成结构化候选时间与理由。
- 固定事件、用户明确指定的时间和不可移动约束是硬约束，模型不能静默覆盖；常识性判断只能作为显式假设，不能伪装成已验证事实。
- 工具负责提供可信忙闲事实和最终冲突校验，不负责替模型选择“什么安排更合理”。

### D. `proposal_item_reschedule`

用户调整提案中某一日程的开始、结束或持续时间，例如“第一个从 10:15 开始，其他没问题”。

- 通过 operation ID 或稳定引用定位单项。
- “从 X 开始”默认保留原持续时间；用户同时给出结束时间或持续时间时以当前消息为准。
- 只读取新旧区间及必要缓冲范围，只重新校验被修改项及与它发生依赖/重叠关系的提案项。
- “其他没问题”显式确认其余 operation 保持不变，不得重新生成其内容。
- “5 点半”这类缺少上午/下午的单点表达不得被静默解释为凌晨。只把原表达、当前本地时间、原日程时间、提前/推迟关系和相邻提案交给一次有界结构化模型推理；日期、持续时间、目标 operation 和冲突校验仍由确定性层掌控。

### Existing adjustment paths

已应用正式日程的修改、Routine 变更、个人占位和非提案型调整继续沿用既有行为。本任务只为“新建行程提案”和“修订仍待确认的提案”增加子意图与快路径；没有有效提案可续接时，安全回退到既有 adjustment 流程或询问一次必要信息。

## Requirements

### R1. Structured continuation

- 每个可供后续修订的提案必须存在服务端结构化载体，不能只有自然语言回复。
- Chat 请求只携带 `conversationId`、父 Run/当前提案引用等不可执行标识；服务端自行读取真实 ChangeSet 和证据，不信任客户端回传 operations。
- `AgentRun` 可追踪父子关系与 continuation 类型；历史 Run 仍可独立查看。

### R2. Stable proposal revision

- ChangeSet operation 具有稳定 `operationId`，在修订版本间保持不变。
- 修订产生新的审计版本；旧待确认提案标记为 superseded，不能与新版本同时显示为可应用。
- 新版本保存 `supersedesChangeSetId` 或等价关系；应用、拒绝、取消均能追溯完整修订链。
- 已应用、已拒绝或已过期提案不能被原地修订。

### R3. Continuation capsule

- 下一轮只注入精简结构化胶囊：提案 ID、父 Run、状态、operation ID、标题/时间/关联 ID、用户已确认部分、证据范围/指纹/时间和未决问题。
- 完整 ToolCall、过程步骤和原始工具 JSON 只用于审计，不进入模型上下文。
- 当结构化胶囊存在时，不重复注入表达同一方案的长助手回复。
- 胶囊序列化上限首发为 1,500 字符；超限时保留目标 operation、相邻依赖和摘要，不任意截断 JSON。

### R4. Evidence freshness and delta validation

- 证据至少包含 `resourceKey`、查询范围、版本/指纹、`observedAt`、生产工具和已验证 operation ID。
- 同一提案、同一窗口指纹且仍在有效期内时，可复用未变 operation 的候选校验。
- 被修改项必须重新校验；若只改变一个区间，不得默认重新校验全部提案。
- 日程窗口已变化或证据过期时，只读取受影响窗口；只有影响无法局部界定时才退回完整窗口。
- 首发默认 TTL 为 5 分钟，但版本/指纹判断优先于 TTL。

### R5. Apply-time safety

- ChangeSet 保存提案覆盖的日程窗口指纹或等价前置条件。
- 用户点击应用时，服务端在写入事务内对全部最终日程操作执行权威冲突与版本校验。
- 已变化时返回结构化 `STALE_PLAN`，正式数据保持不变，并把受影响 operation 交给增量修订路径。
- 新流程不得弱化用户确认、实体归属、日程类型、时区、版本冲突和重复写入保护。

### R6. Fine-grained context plan

- `itinerary_create` 可读取当前完整规划所需目标与日程数据。
- `proposal_patch/reorder/item_reschedule` 默认只读取待确认 ChangeSet、对应行程窗口、证据新鲜度和被修改 operation 的必要关联。
- executions、reviews、rhythmSignals 和无关目标不进入提案修订快路径，除非当前消息明确引入新的决策依据。
- 相同 Run 内同工具、同规范化参数且底层版本未变化时命中只读缓存，不重复访问数据源或扩大证据账本。

### R7. Model reasoning for unspecified reorder times

- `proposal_reorder` 在用户未指定新时间时必须经过一次受 schema 约束的大模型推理；不能用固定规则、数组换位或纯工具代码直接决定最终时间。
- 服务端在调用模型前一次性构造精简 `ReorderContext`：待调整 operations、稳定 operation ID、活动标题/类型/时长、固定/可移动标记、用户已确认约束、受影响窗口、忙闲区间、饭点和可用的轻量节奏偏好。
- 模型输出 `ReorderDecision`：受影响 operation ID、新的起止时间、排序理由、使用的假设、是否仍需澄清；不得输出完整 ChangeSet 或改写未受影响 operation。
- 主推理最多一次；候选验证失败时允许携带结构化冲突结果修正一次。两次均失败则停止并只询问一个取舍问题。
- “银行通常应在白天”等常识可以参与候选排序，但若系统没有营业时间等可靠事实，最终文案必须表述为假设，不能声称已核实。
- 模型产生候选，确定性层负责 schema、operation allowlist、持续时间、硬约束、日程冲突、时区、修订链和应用时安全校验。

### R8. Router and plan

- `IntentResolution` 输出 `adjustmentKind`、`continuationOfRunId/changeSetId`、目标 operation 引用、时间/顺序 patch 和置信度。
- Resolver 必须使用最近对话和结构化提案引用；支持 `10.15`、`10:15`、单点开始时间和自然序号。
- `ExecutionPlan` 对新建行程生成完整取证计划，对提案修订生成 delta plan。
- 低置信只阻塞可能改错对象的步骤，不因安全读取或已知 operation 重复追问。

### R9. User-visible process

- 新建行程继续展示完整规划、冲突检查和待确认草案。
- 修订提案展示“复用上一轮方案”“由小律重新推理受影响片段”“只重新校验受影响时间”“生成修订版”，不伪装成从头规划，也不把模型推理描述成机械交换。
- UI 始终只展示最新可应用版本，并提供“查看上一版本变化”的审计入口；不丢失旧 Run 历史。

### R10. Token and execution budget

- 截图基准场景的第二轮最多 2 次主模型调用、不得读取完整目标/Routine、不得重新校验未变化的三个候选。
- 未指定时间的 `proposal_reorder` 必须有真实模型推理调用证据；不允许为了节省 Token 降级为机械时间交换。
- 该场景总 Token 从 14,326 降至不高于 7,000，降低至少 50%。
- 新增 continuation 固定评测集，分别记录 model calls、input/output tokens、工具调用、复用证据数、失效重读数和端到端成功率。
- 性能优化不得降低 adjustment 成功率、确认合规率、最终冲突安全率或历史可解释性。

### R11. Page-scoped context authority

- `selectedGoalId` 只是页面导航状态；只有 `goal-detail` 页面可把当前目标作为 Agent 隐式上下文。
- Today 首页、目标列表、任务详情、Routine、回顾和设置页面均不得携带旧目标焦点，面板必须明确显示“未关联目标”。
- 该边界必须同时约束快捷提示、能力识别、客户端 business/page payload、服务端规则/模型路由、ContextBuilder 和 fallback，不能只隐藏 UI。
- 离开目标详情时，旧目标对话与摘要移出上下文；旧待确认草案保留可见/可审批，但不得自动续接到新的首页指令。
- 父 Run 引用与草案使用同一 revision 边界；历史 Run 保留可审计，但首页新指令不得挂到旧目标 Run 链。
- 当前未设计 task/routine/schedule block 的隐式实体上下文；未来新增时必须按页面类型建立独立、可见、服务端复核的作用域合同。

## Acceptance Criteria

- [x] “帮我安排今天/未来三天”走 `itinerary_create`，完整读取和校验覆盖所有最终候选，并直接产生待确认 ChangeSet。
- [x] “第一个从 10.15 开始，其他没问题”识别为 `proposal_item_reschedule`，保留原持续时间与其他 operation，只重新校验第一个及其受影响关系。
- [x] “英语学习的开始时间推迟到 5 点半”留在上一份提案修订链，模型收到真实当前本地时间并只返回一个 24 小时制时间；吉他练习和其他 operation 不变。
- [x] “把银行放到阅读前面”识别为 `proposal_reorder`，调用大模型基于精简上下文生成合理时间与理由，只处理受影响片段，不重新读取无关目标或回顾。
- [x] 未指定新时间的重排具有可观测 `ReorderDecision`；固定约束未被覆盖，未经验证的营业时间等信息只作为显式假设。
- [x] 模型首个重排候选冲突时，只把结构化冲突反馈给模型修正一次；仍无解时不创建草案并只询问一个取舍问题。
- [x] “删除第三项，再加一个 30 分钟散步”识别为 `proposal_patch`，其余 operation ID 与字段保持一致。
- [x] 最近对话中的“第一个/其他没问题”能绑定服务端结构化提案；不存在有效提案时只问一次必要问题或安全回退。
- [x] 客户端仍引用待确认 ChangeSet 时，未完全解析的调整在有界续接服务内追问，不回落全量 adjustment 链创建第二份可应用草案。
- [x] 新修订版产生后，旧 ChangeSet 不再可应用，修订链、父 Run、证据来源和用户反馈均可追溯。
- [x] 证据新鲜时跳过完整日程读取；证据失效时只重读受影响窗口；应用前始终执行权威最终校验。
- [x] 用户确认前正式日程零写入；并发修改、过期提案和版本冲突均返回可恢复错误。
- [x] 同一 Run 的完全相同只读请求不会第二次访问数据源。
- [x] 基准第二轮不超过 2 次模型调用和 7,000 Token，安全不变量通过率 100%，重复/未授权写入为 0。
- [x] browser-local 模式保留手动日程能力；无法提供服务端续接时明确降级，不伪装成已复用证据。
- [x] 旧 AgentRun、没有 operation ID 的历史 ChangeSet 和既有正式日程仍可查看、审批或按原流程处理。
- [x] 从某一目标详情返回首页后，面板显示“未关联目标”，规则/模型路由、ContextBuilder、fallback 与后续提案续接均不再收到旧目标上下文；已有草案和正式数据不删除。

## Out of Scope

- 不在本任务中新增通用长期记忆或跨 Conversation 的自动提案续接。
- 不自动修改已应用的正式计划；已应用日程继续走新的 adjustment ChangeSet。
- 不用向量检索替代确定性的 Conversation/Run/ChangeSet 关联。
- 不以扩大上下文窗口、完整回放上一轮工具结果或提高 Token 上限作为解决方案。
- 不用纯规则或工具代码替代未指定时间时的大模型合理性推理。
- 不改变 Routine 重复规则的既有领域模型。

## Notes

- 本任务是 V0.4.1 独立优化项，不与 V0.4.0 主任务归档或发布状态绑定。
- 2026-07-21 已确认：未指定新时间的提案重排由大模型推理合理时间；工具只提供事实与安全校验，不机械决定最终安排。
- 获得实施批准前不修改 Agent Runtime、数据库 schema 或 UI。
