# V0.4.0 Agent 能力评测与效率优化

## Goal

把 Agent 优化从主观体验改进变成“能力可枚举、结果可测量、失败可定位、改动可回归”，并在保持正确性和用户确认边界的前提下缩短执行时间。

## Background

- 当前系统已有多种 Agent 能力、工具调用、执行步骤和持久化追踪，但缺少统一能力目录与基准测评。
- 意图识别与任务拆解的好坏目前难以量化，优化后也缺少稳定回归门槛。
- 工具执行存在可并行机会，但必须先识别数据依赖、写冲突和用户确认边界。

## Requirements

### R1. 能力清单

- 从运行时、能力配置、提示词、工具注册表、ChangeSet、API 与 UI 追踪中梳理真实能力。
- 对每项能力记录触发意图、必要上下文、工具、输出、确认要求、预算、失败模式和可观测证据。
- 明确 Agent Loop 与确定性工作流边界。

### R2. QA 评测反馈闭环

- 建立版本化测试集，覆盖正常、模糊、缺信息、冲突、失败恢复和越权边界。
- 定义意图识别、参数提取、任务拆解、工具选择、执行结果、确认合规、时延与成本指标。
- 支持自动评分、人工复核、失败归因、基线对比和回归阻断。
- 首发采用“PR 必跑确定性离线集 + 发布前真实模型抽样 + 失败样本人工复核”，真实模型波动不得替代稳定门禁。

### R3. 意图识别与任务拆解

- 优先从对话与产品上下文提取信息，只在仍缺少阻塞字段时提问。
- 对多意图、隐含意图、歧义、跨轮修正和不可执行请求定义决策行为。
- 任务拆解应显式表达依赖、可并行组、确认点、完成条件和失败恢复策略。
- V0.4.0 支持识别多意图并在一个父运行中形成有序步骤；不建设通用多 Agent 编排平台。

### R4. 执行效率

- 建立端到端与分阶段耗时基线。
- 对无依赖的只读工具、互不冲突的独立查询启用受控并行。
- 对写操作、依赖前序输出、共享资源或需要用户确认的步骤保持串行或设立屏障。
- 并行失败必须可追踪、可部分重试，不得导致重复写入或错误完成判定。
- 首发并行范围限定为独立上下文读取和模型同批给出的独立 read 工具；draft_write、正式写入及依赖前序输出的工具保持串行。

## Acceptance Criteria

- [x] 形成与当前代码一一对应的 Agent 能力矩阵，不包含无法在实现中找到证据的“想象能力”。
- [x] 每项核心能力至少有正常、歧义、缺失信息、工具失败和安全边界用例。
- [x] 测评输出能定位失败阶段，并能比较优化前后准确率、任务成功率、P50/P95 时延、工具调用数与 token 消耗。
- [x] 意图识别和任务拆解具有明确的结构化中间结果或等价可观测证据。
- [x] 并行调度器只并行满足依赖与冲突规则的步骤，并通过测试证明不会重复写入。
- [x] 性能优化不得降低任务成功率、确认合规率或执行历史可解释性。
- [x] Router 固定集 overall top-1 ≥ 90%、单能力 ≥ 85%、multi-intent recall ≥ 85%、slot F1 ≥ 90%。
- [x] 安全/确认/冲突/重复写入不变量通过率为 100%，相对基线任务成功率下降不超过 2 个百分点。
- [x] 上下文构建 P50 至少降低 30%，符合并行条件的工具批次 P95 wall time 至少降低 25%。

## Out of Scope

- 不以单一模型更换代替能力评测体系。
- 不为了速度绕过 ChangeSet、用户确认、幂等性或持久化追踪。

## Confirmed Decisions

- 2026-07-20：首发基线为 Router 120 例、Planner 30 例、Runtime 30 例、Performance 10 例；PR 跑确定性集，发布前跑真实模型抽样，失败样本人工复核。
