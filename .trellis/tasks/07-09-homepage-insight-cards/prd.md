# 首页右侧建议卡片

> 父任务：`07-09-v03-release`  
> 产品文档：`docs/v0.3.0 版本需求/新增特性-首页右侧建议卡片.md`  
> 状态：Phase B 已完成；**Phase B.1 刷新策略变更** 规划中

## 目标

将首页右侧三张卡片改为基于真实执行数据的 **Action → Insight → Progress** 洞察，通过 **Facts + LLM** 生成可执行建议，并支持建议追踪。

## 阶段划分

| 阶段 | 内容 | 状态 |
|------|------|------|
| **Phase A** | 客户端规则 + 轻量调整按钮 | 已完成 |
| **Phase B** | Facts + LLM + 快照落库 + 前端接入 | 已完成 |
| **Phase B.1** | **定时 + 手动** 刷新策略（替代哈希失效） | 已实现 |
| **Phase C** | 完整 `Suggestion` 实体与北极星指标 | 后续 |

## Phase B.1 变更背景（2026-07-09）

**问题**：`factsHash` 判定过期不稳定——慢卡字段随时间自然漂移（missed、executionByPeriod、signal 过期），导致刷新时常重算；AI 生成成功后易被规则降级覆盖。

**决策**：废弃「哈希变化 → 自动重生成」；改为 **定时入库 + 读最新记录 + 手动刷新**。

## Phase B.1 已确认需求

### 刷新节奏

| 卡片 | 定时策略 | 说明 |
|------|----------|------|
| **此刻建议**（moment） | **每小时** 生成一次并入库 | 按用户时区整点（`:00`） |
| **节奏发现 + 本周轨道**（slow，一次 LLM 输出两卡） | **每周 2 次**：**周三 08:00**、**周日 20:00**（用户时区） | 时间点由实现方选定，已确认 |
| **三张卡** | 均支持 **手动点击更新** | 立即触发对应路径重新生成 |

### 读取策略

- 用户进入首页：`GET /api/home/insights` **只读各 kind 最新一条**快照，不在 GET 时因哈希/TTL 自动重算。
- **冷启动**：若某 kind 尚无快照，GET 时可 **按需生成一次**（避免白屏），之后走定时/手动。

### 存储与清理

- 快照 **追加写入**（保留历史 `generatedAt`），读取时取最新。
- 定期清理过久记录（具体保留窗口见 Open Questions）。

### 上下文与生成逻辑

- slow（节奏发现 + 本周轨道）注入上下文与 Phase B **基本一致**（`slowFacts` + 现有 prompt）。
- moment 上下文与 Phase B **基本一致**（`momentFacts` + 现有 prompt）。
- 「换个建议」仍轮换 `alternateCandidates`，不调 LLM。

### 废弃行为

- ❌ `factsHash` 不匹配 → 自动重生成
- ❌ `validUntil` TTL → 自动重生成
- ❌ 日程/执行写入 → `invalidateMomentInsights`
- ❌ Review 生成 → `invalidateSlowInsights`

（日程变化的影响由 **下一次定时或手动刷新** 体现。）

### 本地模式

- 无 DB：继续 Phase A 客户端规则，不调 insight API 生成。

## 验收标准（Phase B.1）

- [ ] 刷新页面后，三卡文案与来源在定时周期内 **保持不变**（除非手动刷新或到达定时点）
- [ ] moment 在用户时区每小时整点由 cron 生成新记录
- [ ] slow 在用户时区周三 08:00、周日 20:00 由 cron 生成新记录
- [ ] 每张卡有手动「更新」入口，点击后生成新记录并更新 UI
- [ ] GET 仅返回最新快照；`meta` 暴露各卡 `generatedAt` 与是否刚手动/定时生成
- [ ] 历史快照按保留策略自动清理
- [ ] 需求文档 §6.2 与实现一致
