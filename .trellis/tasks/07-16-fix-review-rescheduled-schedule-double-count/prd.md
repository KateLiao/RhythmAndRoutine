# 修复回顾重复计算改期日程

## Goal

回顾模块计算周期日程时，只消费最终确定、仍代表当前日历安排的日程块，避免改期前后两份持久化记录被重复计入，同时保留完整改期历史以供审计和追溯。

## Confirmed Facts

- `ScheduleBlock` 通过 `rescheduledFromId` 形成改期链；统一改期事务会把原块保留为 `RESCHEDULED`，并新建指向原块的后继块。
- 日历通过 `isActiveCalendarBlock` 隐藏被替代的普通改期历史块。
- `src/server/services/reviews.ts` 的 `fetchPeriodBlocks` 当前按周期直接读取全部未软删除的 `ScheduleBlock`，导致改期前的旧块进入周期总数、每日聚合、偏差、周摘录和 LLM facts。
- 回顾后续所有确定性计算都消费 `fetchPeriodBlocks` 的结果，因此应在该数据入口统一消除被后继块替代的历史行。

## Requirements

- R1. 数据层继续保留改期前后的全部 `ScheduleBlock`，不得删除或覆盖历史记录。
- R2. 回顾查询必须识别周期内日程块是否已有 `rescheduledFromId` 指向它的后继块；已有后继的块视为被替代，不得进入回顾计算。
- R3. 单次改期与连续多次改期都只能计算改期链的最终叶子节点；链中每个已被后继替代的节点都不得计入。
- R4. 最终叶子节点是否属于某个回顾周期，以该叶子节点自身的 `startsAt` 为准。若旧块在本周期、最终块在其他周期，本周期不得保留旧块作为替代计数。
- R5. 取消日程、未完成日程以及没有后继记录的既有 `RESCHEDULED` 数据保持当前回顾语义，本次不扩大为状态规则调整。
- R6. 过滤必须发生在回顾 facts 的统一入口，确保指标、日聚合、任务/目标进展、偏差摘要、代表性摘录和 LLM 输入使用同一份最终日程集合。
- R7. 用户点击手动生成或重新生成后，页面必须立即展示明确的进行中反馈，并在请求完成前禁用重复触发；请求失败后恢复按钮并给出可理解的重试提示。
- R8. 桌面端日回顾的「收工复盘」侧栏应获得更宽的弹性占比，改善长文本阅读；周回顾现有比例和 1050px 以下单列布局保持不变。

## Acceptance Criteria

- [x] 一条日程改期一次且新旧块均落在同一回顾周期时，回顾只计算新块一次。
- [x] 一条日程连续改期两次时，回顾只计算最后一个块，两个历史块均排除。
- [x] 改期后的最终块落在周期外时，原周期不会继续计算已被替代的旧块。
- [x] 数据库中的改期链历史记录不被修改或删除。
- [x] 非改期块和没有后继的块继续按原规则进入回顾。
- [x] 新增自动化回归测试覆盖过滤规则；相关测试、类型检查与 lint 通过。
- [x] 点击重新生成后，按钮立即显示旋转进度图标与“正在重新生成…”文案，并在请求期间不可再次点击。
- [x] 生成成功后展示既有成功通知；生成失败后按钮恢复并显示重试提示。
- [x] 桌面端日回顾的「收工复盘」占据约四成内容宽度；周回顾和窄屏布局不回退。

## Out of Scope

- 不改变改期的写入模型、日历展示组件或数据库 schema。
- 不重算或迁移已经生成并持久化的历史 Review。
- 不改变取消日程和无后继异常数据的现有回顾口径。
